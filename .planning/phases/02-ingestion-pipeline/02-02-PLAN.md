    Create src/prme/ingestion/schema.py with Pydantic models for LLM extraction output:

    1. ExtractedEntity(BaseModel):
       - name: str = Field(description="Entity name as it appears in the text")
       - entity_type: str = Field(description="One of: person, organization, location, product, concept, event")
       - description: str | None = Field(default=None, description="Brief contextual description from the text")

    2. ExtractedFact(BaseModel):
       - subject: str = Field(description="Entity this fact is about")
       - predicate: str = Field(description="Relationship or attribute type (e.g., works_at, lives_in, role)")
       - object: str = Field(description="Value or target entity")
       - confidence: float = Field(default=0.5, ge=0.0, le=1.0, description="Extraction confidence")
       - temporal_ref: str | None = Field(default=None, description="Raw temporal reference if mentioned (e.g., 'yesterday', 'last month')")
       - fact_type: str = Field(default="fact", description="One of: fact, decision, preference")

    3. ExtractedRelationship(BaseModel):
       - source_entity: str = Field(description="Source entity name")
       - target_entity: str = Field(description="Target entity name")
       - relationship_type: str = Field(description="Edge type: relates_to, part_of, caused_by, supports, mentions")
       - confidence: float = Field(default=0.5, ge=0.0, le=1.0)

    4. ExtractionResult(BaseModel):
       - entities: list[ExtractedEntity] = Field(default_factory=list)
       - facts: list[ExtractedFact] = Field(default_factory=list)
       - relationships: list[ExtractedRelationship] = Field(default_factory=list)
       - summary: str | None = Field(default=None, description="Brief summary of the message content")

    Note: fact_type field on ExtractedFact allows single-pass extraction of facts, decisions, AND preferences per user decision. The LLM classifies each triple as one of these types. This maps to NodeType.FACT, NodeType.DECISION, or NodeType.PREFERENCE during materialization.

    Create src/prme/ingestion/extraction.py:

    1. EXTRACTION_SYSTEM_PROMPT constant: Detailed instructions for the LLM to extract:
       - Named entities (people, organizations, locations, products, concepts)
       - Facts as subject-predicate-object triples with confidence scores
       - Decisions (fact_type="decision") and preferences (fact_type="preference") as special fact types
       - Relationships between entities
       - A brief summary of the message
       - Temporal references (include raw text like "yesterday", "last week")
       - Instruction: Only extract information explicitly stated or strongly implied by the text. Do NOT infer facts not grounded in the source.

    2. ExtractionProvider Protocol (runtime_checkable):
       - @property provider_name -> str
       - @property model_name -> str
       - async def extract(self, content: str, *, role: str = "user") -> ExtractionResult

    3. InstructorExtractionProvider class:
       - __init__(provider_string: str, *, model: str | None = None, max_retries: int = 3, timeout: float = 30.0)
         - Store provider_string (e.g., "openai/gpt-4o-mini", "anthropic/claude-3-5-sonnet-20241022", "ollama/llama3.2")
         - Lazy client initialization: _client = None, created on first extract() call
       - _ensure_client(): Lazily create instructor client via instructor.from_provider(self._provider_string, async_client=True). This avoids API key validation at construction time (Pitfall 4 from research).
       - provider_name property: return provider_string.split("/")[0]
       - model_name property: return provider_string
       - async extract(content, *, role="user") -> ExtractionResult:
         - Call self._ensure_client()
         - Use client.create(response_model=ExtractionResult, messages=[system prompt, user message], max_retries=self._max_retries)
         - Return the result
         - On exception: log error, return empty ExtractionResult (fail open -- pipeline handles retry)

    4. create_extraction_provider(config: ExtractionConfig) -> ExtractionProvider factory function:
       - Build provider_string from config: f"{config.provider}/{config.model}"
       - Return InstructorExtractionProvider(provider_string, max_retries=config.max_retries, timeout=config.timeout)

    Create src/prme/ingestion/__init__.py:
    - Re-export: ExtractionResult, ExtractedEntity, ExtractedFact, ExtractedRelationship from schema
    - Re-export: ExtractionProvider, InstructorExtractionProvider, create_extraction_provider from extraction

    Create src/prme/ingestion/grounding.py implementing source text validation to filter LLM hallucinations:

    1. validate_grounding(result: ExtractionResult, source_text: str) -> ExtractionResult:
       - source_lower = source_text.lower()
       - Filter entities: keep only if entity.name.lower() is a substring of source_lower. Log discarded entities at WARNING level.
       - Filter facts: keep only if fact.subject.lower() is a substring of source_lower. The subject must be grounded (it references an entity name from the text). Object does not need to be a substring -- it may be an attribute value like "senior engineer" that is paraphrased. Log discarded facts.
       - Filter relationships: keep only if BOTH source_entity.lower() AND target_entity.lower() are substrings of source_lower. Both endpoints must be grounded. Log discarded relationships.
       - Summary is always kept (it's a paraphrase, not an extraction).
       - Return new ExtractionResult with filtered lists.

    2. Use structlog for logging (consistent with project patterns). Log at WARNING level for each discarded item with the item name/details and the reason (not grounded in source).

    Per user decision: "Validate extracted entities/facts against source text -- discard ungrounded/hallucinated extractions." Per research: start with substring match (conservative). Better to discard valid context-dependent references than accept hallucinations.

    Add grounding re-exports to src/prme/ingestion/__init__.py: validate_grounding.
