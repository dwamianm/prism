# Phase 3.4: Namespace & Config Expansion - Research

**Researched:** 2026-03-01
**Domain:** Python enum expansion, pydantic-settings nested config, [HYPOTHESIS] parameter wiring
**Confidence:** HIGH

## Summary

Phase 3.4 is a well-scoped refactoring and configuration expansion phase. It has two distinct work streams: (1) expanding the `Scope` enum from 3 members to 6 and renaming `ORG` to `ORGANISATION`, then updating all ~16 source files plus ~4 test files that reference it; and (2) nesting `ScoringWeights`, `PackingConfig`, and confidence-related parameters under `PRMEConfig` as configurable sub-fields, wiring all `[HYPOTHESIS]`-marked parameters through the config system.

The codebase already uses pydantic-settings with `env_prefix="PRME_"` and `env_nested_delimiter="__"` on `PRMEConfig`. Nesting `ScoringWeights` and `PackingConfig` under it is straightforward -- they are already Pydantic BaseModel instances with correct field definitions. The confidence matrix override requires a dict merge pattern. The `EPISTEMIC_WEIGHTS` dict and `UNVERIFIED_CONFIDENCE_THRESHOLD` constant are currently module-level globals that need to be elevated into `PRMEConfig`.

No new external libraries are needed. All work uses existing pydantic (>=2.12) and pydantic-settings (>=2.9) capabilities. The risk is primarily in the rename blast radius (`ORG` -> `ORGANISATION`) and ensuring the config wiring doesn't break the module-level `DEFAULT_*` constant pattern that existing code depends on.

**Primary recommendation:** Tackle the enum expansion first (clean break, update all references), then the config nesting second (add sub-fields to PRMEConfig, wire defaults), then the [HYPOTHESIS] parameter collection last (confidence overrides, epistemic weights, unverified threshold).

### Pattern 1: Enum Expansion with Clean Break
**What:** Rename `ORG` to `ORGANISATION`, add `AGENT`, `SYSTEM`, `SANDBOX` to the `Scope` enum. No aliases, no backward compatibility shim.
**When to use:** Pre-release project with no production data (confirmed in CONTEXT.md).
**Example:**
```python
class Scope(str, Enum):
    """Memory scoping levels.

    All six RFC-0004 namespace types supported.
    """
    PERSONAL = "personal"
    PROJECT = "project"
    ORGANISATION = "organisation"  # Renamed from ORG per RFC-0004
    AGENT = "agent"               # Per-agent working memory
    SYSTEM = "system"             # System-generated content (summaries, organizer output)
    SANDBOX = "sandbox"           # Temporary isolated scope; supports HARD_DELETE expiry action
```

### Pattern 2: Nested Config with Defaults Fallback
**What:** Add `ScoringWeights` and `PackingConfig` as sub-fields on `PRMEConfig`. Module-level `DEFAULT_*` constants remain as fallbacks when no config is provided.
**When to use:** When existing code uses module-level defaults that must continue working without a PRMEConfig instance.
**Example:**
```python
class PRMEConfig(BaseSettings):
    # ... existing fields ...
    scoring: ScoringWeights = Field(
        default_factory=ScoringWeights,
        description="Scoring weights for composite retrieval formula",
    )
    packing: PackingConfig = Field(
        default_factory=PackingConfig,
        description="Context packing configuration",
    )
    # Env vars: PRME_SCORING__W_SEMANTIC=0.30, PRME_PACKING__TOKEN_BUDGET=4096
    model_config = {
        "env_prefix": "PRME_",
        "env_nested_delimiter": "__",
    }
```

**Critical detail:** `ScoringWeights` is currently a frozen `BaseModel`, not a `BaseSettings`. pydantic-settings `env_nested_delimiter` works with nested BaseModel fields on a root BaseSettings class. The nested model does NOT need to be BaseSettings itself -- the root BaseSettings handles env var parsing and delegates to nested models. This is the established pydantic-settings pattern and matches how `EmbeddingConfig` and `ExtractionConfig` already work in this codebase (though those are currently BaseSettings subclasses, they could equally be BaseModel).

**Important:** ScoringWeights currently uses `ConfigDict(frozen=True)`. When nested under PRMEConfig, pydantic-settings will construct it from env vars at startup and the frozen constraint applies after construction. The model_validator sum check runs during construction, so env var values that don't sum to 1.0 will correctly fail at startup. This is the desired behavior per CONTEXT.md.

### Pattern 3: Confidence Matrix Override Merge
**What:** `PRMEConfig.confidence_overrides` is a `dict[str, float]` with keys like `"observed:user_stated"`. At startup, the overrides merge into `DEFAULT_CONFIDENCE_MATRIX` to produce the active matrix.
**When to use:** When deployers need to change a few matrix cells without re-specifying all ~20 entries.
**Example:**
```python
class PRMEConfig(BaseSettings):
    confidence_overrides: dict[str, float] = Field(
        default_factory=dict,
        description=(
            "Override specific confidence matrix cells. "
            "Keys: 'epistemic_type:source_type' (e.g., 'observed:user_stated'). "
            "Values: float 0.0-1.0. Merges into default matrix."
        ),
    )

    @model_validator(mode="after")
    def validate_confidence_overrides(self) -> "PRMEConfig":
        for key, value in self.confidence_overrides.items():
            if not (0.0 <= value <= 1.0):
                raise ValueError(
                    f"Confidence override '{key}' = {value} out of range [0.0, 1.0]"
                )
            parts = key.split(":")
            if len(parts) != 2:
                raise ValueError(
                    f"Confidence override key '{key}' must be 'epistemic_type:source_type'"
                )
        return self
```

Then on `ConfidenceMatrix`:
```python
class ConfidenceMatrix(BaseModel):
    # ... existing ...

    def with_overrides(self, overrides: dict[str, float]) -> "ConfidenceMatrix":
        """Return a new ConfidenceMatrix with overrides merged in."""
        merged = dict(self.matrix)
        for key, value in overrides.items():
            epistemic, source = key.split(":")
            merged[(epistemic, source)] = value
        return ConfidenceMatrix(matrix=merged)
```

**Example:**

### Anti-Patterns to Avoid
- **Dual source of truth for defaults:** Do NOT duplicate default values in both PRMEConfig field defaults and module-level `DEFAULT_*` constants. The module-level constants should construct from default PRMEConfig, or the config fields should reference the same default factory. Pick one canonical source.
- **BaseSettings on nested models:** Do NOT make ScoringWeights or PackingConfig into BaseSettings subclasses. They are nested models under the root PRMEConfig BaseSettings. Making them BaseSettings too would cause double env-var loading with conflicting prefixes.
- **Migration code for ORG->ORGANISATION:** Do NOT add aliases, backward-compat shims, or migration logic. This is pre-release code with no production data. Clean break per CONTEXT.md decision.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Env var parsing for nested config | Custom env var parser | pydantic-settings `env_nested_delimiter="__"` | Already configured in PRMEConfig; handles type coercion, nested objects, defaults |
| Config validation | Manual value checking | Pydantic `model_validator`, `Field(ge=, le=)` | ScoringWeights sum check is already a model_validator; confidence range is already `ge=0.0, le=1.0` |
| Dict merge for overrides | Custom deep-merge utility | Simple dict.update() with new ConfidenceMatrix construction | The matrix is flat (no nesting), so shallow merge is correct |

**Key insight:** pydantic-settings already handles the hard parts (env var parsing, type coercion, nested delimiter splitting, .env file loading). The phase work is primarily about wiring existing Pydantic models into the existing BaseSettings root, not building new config infrastructure.

## Common Pitfalls

### Pitfall 1: Breaking the Module-Level Default Pattern
**What goes wrong:** Existing code imports `DEFAULT_SCORING_WEIGHTS` and `DEFAULT_PACKING_CONFIG` from `retrieval/config.py`, and `DEFAULT_CONFIDENCE_MATRIX` from `epistemic/matrix.py`. If these constants are removed or changed to require a PRMEConfig instance, all existing call sites break.
**Why it happens:** Eager refactoring removes the constants in favor of config-only access.
**How to avoid:** Keep `DEFAULT_*` constants as module-level singletons constructed from default values. When a `PRMEConfig` instance is available (in MemoryEngine, RetrievalPipeline), use config values. When not available (standalone function calls, tests), fall back to module-level defaults. This is the established pattern.
**Warning signs:** Import errors or missing default argument values in function signatures.

### Pitfall 2: ScoringWeights Frozen + Env Var Construction Order
**What goes wrong:** ScoringWeights uses `ConfigDict(frozen=True)` which prevents mutation after construction. If pydantic-settings tries to set individual fields after construction, it fails.
**Why it happens:** Misunderstanding of when frozen takes effect.
**How to avoid:** pydantic-settings constructs the nested model atomically from parsed env vars. The frozen constraint applies after the model is fully constructed. The model_validator runs during construction. This is correct behavior -- no workaround needed. Verified by the existing pattern where `EmbeddingConfig` nested under `PRMEConfig` already works.
**Warning signs:** `ValidationError: Instance is frozen` during env var loading. (Should not happen with correct nesting.)

### Pitfall 3: LLM Extraction Prompt Scope Values
**What goes wrong:** The LLM extraction system prompt in `extraction.py` (line 58) and field descriptions in `schema.py` (lines 29, 65) hardcode `"org"` as a valid scope value. After renaming to `"organisation"`, the LLM will still extract `"org"` from conversations, causing scope mapping failures.
**Why it happens:** Prompts are string literals, not generated from the enum.
**How to avoid:** Update all prompt strings and Field descriptions to use `"organisation"`. Also update the ingestion pipeline scope mapping logic (`_parse_scope()` or equivalent) to handle the new value.
**Warning signs:** Extracted entities/facts with null scope when the LLM outputs the old `"org"` value.

### Pitfall 4: Test File References to Scope.ORG
**What goes wrong:** Tests in `test_retrieval_filter_forwarding.py` (line 292) use `Scope.ORG`. If the enum member is renamed but tests are not updated, they fail with AttributeError.
**Why it happens:** Tests are not in the same file list as source code during refactoring.
**How to avoid:** The grep output shows exactly which test files reference `Scope.ORG`: `test_retrieval_filter_forwarding.py` line 292. Update all test files in the same commit as the enum rename.
**Warning signs:** `AttributeError: 'Scope' object has no attribute 'ORG'` in test output.

### Pitfall 5: Confidence Override Env Var Complexity
**What goes wrong:** Dict-type env vars are complex to specify. `PRME_CONFIDENCE_OVERRIDES` would need to be JSON-encoded in the environment, which is error-prone.
**Why it happens:** pydantic-settings parses dict fields from env vars as JSON strings.
**How to avoid:** Accept that confidence_overrides is primarily a programmatic config (passed in code) rather than an env-var config. Document this clearly. Env var support works via JSON string (`PRME_CONFIDENCE_OVERRIDES='{"observed:user_stated": 0.95}'`), but most users will set it in code or a config file. Same applies to epistemic_weights dict.
**Warning signs:** Users reporting JSON parse errors when trying to set dict overrides via env vars.

### ConfidenceMatrix.with_overrides()
```python
# src/prme/epistemic/matrix.py (addition)
def with_overrides(self, overrides: dict[str, float]) -> "ConfidenceMatrix":
    """Return a new ConfidenceMatrix with specified cells overridden.

    Overrides merge into the existing matrix. New keys (not in the
    default matrix) are accepted for forward-compatibility.

    Args:
        overrides: Dict of "epistemic_type:source_type" -> confidence.

    Returns:
        New ConfidenceMatrix instance with overrides applied.
    """
    if not overrides:
        return self
    merged = dict(self.matrix)
    for key, value in overrides.items():
        epistemic, source = key.split(":")
        merged[(epistemic, source)] = value
    return ConfidenceMatrix(matrix=merged)
```

## Open Questions

1. **TOML/YAML config file support**
   - What we know: pydantic-settings supports TOML via `pydantic-settings[toml]` extra and YAML via custom source. The project is an embeddable library, not a service.
   - What's unclear: Whether deployers would benefit from file-based config vs programmatic + env vars.
   - Recommendation: **Skip for Phase 3.4.** Env vars + programmatic config is sufficient. A `from_file()` convenience can be added later without breaking changes. The dict fields (confidence_overrides, epistemic_weights) are easier to specify in code than in env vars, and that's fine for an embeddable library where the consumer writes Python code.

2. **Config logging at DEBUG level**
   - What we know: Logging active config at startup helps debugging. PRMEConfig values may contain sensitive paths.
   - What's unclear: Whether deployers want this by default.
   - Recommendation: **Add a single DEBUG log line in MemoryEngine.create()** logging the config summary (scoring version_id, packing token_budget, number of confidence overrides). Not the full config object. Lightweight and useful.

3. **SANDBOX/HARD_DELETE relationship encoding**
   - What we know: RFC-0004 S7 says HARD_DELETE is permitted ONLY for SANDBOX namespaces.
   - What's unclear: Best way to encode this constraint when namespace behavior is not implemented in this phase.
   - Recommendation: **Docstring note on the SANDBOX enum member** stating it supports HARD_DELETE. A module-level constant `HARD_DELETE_ALLOWED_SCOPES = frozenset({Scope.SANDBOX})` is also clean and useful for future enforcement, but not required in this phase since NSPC-04 (retention policy) is deferred to Phase 5.

4. **Matrix override env vars**
   - What we know: `PRME_CONFIDENCE_OVERRIDES='{"observed:user_stated": 0.95}'` works but is awkward.
   - What's unclear: Whether deployers will primarily use env vars or programmatic config for overrides.
   - Recommendation: **Document both paths.** Env var JSON for simple cases, programmatic `PRMEConfig(confidence_overrides={...})` for complex cases. Do not add individual env vars per matrix cell (explosion of env var names).
