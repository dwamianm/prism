## Summary

This phase is a documentation reconciliation task, not a code implementation task. The Revised RFC suite (RFC-0000 through RFC-0014) was added to the project after Phases 1-2 were planned and built. The current REQUIREMENTS.md was derived from the original spec documents (RFC_0001_PRME_Technical_Spec, RFC_0002_RMS_Git_Sync_Profile). The Revised RFCs introduce significant new concepts, terminological changes, additional node/edge types, epistemic modeling, a full operation log schema, expanded namespace model, and multiple new lifecycle/scoring subsystems not captured in current requirements.

The reconciliation requires systematic comparison of each Revised RFC against current REQUIREMENTS.md, identifying: (1) new requirements not present at all, (2) existing requirements that need updating to match revised spec language, and (3) deltas between built code (Phases 1-2.2) and revised specs that require either immediate fixes or deferred work items.

**Primary recommendation:** Work through all 15 RFCs in numerical order (as per user decision), producing an updated REQUIREMENTS.md with new requirement IDs and inline RFC source references, plus a separate delta report documenting code-vs-spec gaps with triage recommendations.

### Pattern 1: Per-RFC Review Checklist

**What:** Process each RFC sequentially, extracting requirements and comparing against REQUIREMENTS.md
**When to use:** For every RFC from 0000 through 0014
**Structure:**
```
For each RFC-XXXX:
  1. Read the RFC conformance requirements section
  2. Compare against existing REQUIREMENTS.md entries
  3. Identify: new requirements, updated requirements, unchanged requirements
  4. For new requirements: assign ID with new prefix, add source reference
  5. For updated requirements: modify in-place, add change note
  6. For built features (Phases 1-2.2): check code against RFC MUST requirements
```

### Pattern 2: Delta Triage Classification

**What:** Classify each code-vs-spec gap by severity and recommended action
**Recommended scheme (3-tier):**

| Tier | Label | Criteria | Action |
|------|-------|----------|--------|
| P0 | FIX NOW | Delta in built feature that violates a MUST requirement from completed-phase RFCs (0001-0004) | Create gap-closure phase |
| P1 | FIX BEFORE PHASE | Delta in built feature that affects upcoming phase (e.g., Phase 3 retrieval needs a field that Phase 1 models don't have) | Fix before the dependent phase |
| P2 | DEFER | Delta in unbuilt feature, or SHOULD/MAY-level gap in built feature | Track in requirements, fix in natural phase order |

## Common Pitfalls

### Pitfall 1: Missing the Forest for the Trees
**What goes wrong:** Getting lost in individual RFC details and failing to spot systemic changes that affect multiple requirements
**Why it happens:** The RFC suite has cross-cutting concerns (e.g., `[HYPOTHESIS]` markers, scoped determinism, policy versioning) that appear in every RFC
**How to avoid:** First pass: read all 15 RFCs and identify cross-cutting themes. Second pass: map requirements. This prevents discovering on RFC-0014 that a pattern from RFC-0002 was missed
**Warning signs:** Discovering the same pattern for the Nth time late in the review

### Pitfall 2: Overclassifying Deltas as P0
**What goes wrong:** Every difference between built code and RFC spec gets flagged as "fix now," creating an unmanageable list of gap-closure work
**Why it happens:** Natural tendency to want everything perfect
**How to avoid:** Use the 3-tier triage strictly. P0 is ONLY for MUST requirements in already-built phases that are actively violated. SHOULD-level gaps in built code are P1 at most
**Warning signs:** More than 10 P0 items -- probably over-classifying

### Pitfall 3: Confusing RFC Tier Requirements with Current Phase Scope
**What goes wrong:** Adding Tier 4 requirements (multi-agent, branching) as if they need to be built in the next phase
**Why it happens:** The RFCs state "MUST" for their tier, but those tiers map to future phases
**How to avoid:** Map RFC tiers to roadmap phases. Tier 0-1 requirements affect built code. Tier 2+ requirements are for future phases. All get documented, but only Tier 0-1 deltas produce P0 triage items
**Warning signs:** P0 items referencing RFC-0010 through RFC-0014

### Pitfall 4: Terminology Drift Between Original and Revised Specs
**What goes wrong:** The original spec uses "Tentative/Stable/Superseded/Archived" lifecycle states. The Revised RFCs use "ACTIVE/DEPRECATED/SUPERSEDED/ARCHIVED." Requirements use the old terminology. Code uses the old terminology. Reconciliation fails to notice the terminology change.
**Why it happens:** The change is subtle -- the states serve similar purposes but have different names and different semantics
**How to avoid:** Create an explicit terminology mapping table as the first reconciliation step
**Warning signs:** Requirements that reference lifecycle states without checking which terminology the code actually uses

### Pitfall 5: Losing Track of Original RFC Content That Was Dropped
**What goes wrong:** The Revised RFCs dropped some original content (e.g., emotional signal tracking from original RFC-0013). If not noticed, orphaned requirements may remain in REQUIREMENTS.md
**Why it happens:** Easier to spot additions than deletions
**How to avoid:** Cross-reference original RFCs against Revised suite per user decision. Mark anything dropped as explicitly removed with rationale
**Warning signs:** Requirements in REQUIREMENTS.md that have no matching section in any Revised RFC

### Pitfall 6: Contradictions with STATE.md Project Decisions
**What goes wrong:** A Revised RFC specifies something that contradicts a decision recorded in STATE.md (e.g., the RFC says lifecycle states are ACTIVE/DEPRECATED but the project decided on Tentative/Stable). Auto-resolving in either direction breaks something.
**Why it happens:** STATE.md records implementation decisions; RFCs record spec requirements. They can diverge.
**How to avoid:** Per user instruction, flag these for manual review. Don't auto-resolve. Document both the RFC requirement and the existing decision, and note the conflict in the delta report.
**Warning signs:** Updating a requirement that contradicts a bullet in STATE.md

## Detailed RFC-by-RFC Analysis

This section catalogs the key findings from reading all 15 Revised RFCs plus the 2 original RFCs. This is the core research the planner needs.

### RFC-0000: Suite Overview and Design Philosophy
**Type:** Informational / cross-cutting
**Impact on REQUIREMENTS.md:** Introduces 8 design principles (P1-P8) not captured as requirements. Introduces conformance tiers and `[HYPOTHESIS]` marker convention. Introduces suite terminology table.
**New concepts not in current requirements:**
- Scoped determinism (P2): strict at storage/derivation, best-effort at extraction
- Cost awareness (P5): every memory object has a retrieval cost measured in tokens
- Empirical humility (P6): unvalidated parameters marked `[HYPOTHESIS]`
- Fail safely (P8): degrade to no-memory, not incorrect-memory
**Delta with built code:** Principles are architectural, not directly testable. Current code generally aligns with P1 (append-only), P2 (scoped determinism via design decisions), P7 (portability intent). No code delta to report.
**Recommendation:** Add a "Design Principles" section to REQUIREMENTS.md documenting P1-P8 with RFC-0000 references. No P0 items.

### RFC-0001: Core Data Model and Terminology
**Type:** Tier 0 -- Foundation
**Impact on REQUIREMENTS.md:** Defines the canonical data model. Significant terminology and structural changes from current code.
**Key deltas with built code:**

1. **Event model fields:** RFC-0001 requires `stream`, `actor_id`, `actor_type`, `role` (expanded set: USER/ASSISTANT/TOOL/SYSTEM/ORGANISER), `namespace_id`, `content_hash`, `sequence_num`, `prev_hash`. Current Event model has: `user_id`, `session_id`, `role` (user/assistant/system), `content`, `content_hash`, `metadata`. Missing: `stream`, `actor_id`, `actor_type`, `namespace_id`, `sequence_num`, `prev_hash`.

2. **Memory object fields:** RFC-0001 requires `version`, `epistemic_type`, `source_type`, `evidence_ids`, `asserted_by`, `valid_from`, `valid_to`, `created_at`, `last_modified_at`, `lifecycle_state`, `superseded_by`, `confidence`, `salience`, `namespace_id`, `value`, `structured_value`. Current MemoryNode has most of these but is missing: `version`, `epistemic_type`, `source_type`, `asserted_by`, `namespace_id`, `value` (uses `content` instead), `structured_value`.

3. **Lifecycle states:** RFC-0001 uses ACTIVE/DEPRECATED/SUPERSEDED/ARCHIVED. Built code uses TENTATIVE/STABLE/SUPERSEDED/ARCHIVED. This is a semantic and naming mismatch. TENTATIVE->STABLE promotion is not in the RFC lifecycle. The RFC's ACTIVE encompasses both TENTATIVE and STABLE.

4. **Node types:** RFC-0001 defines FACT, PREFERENCE, DECISION, TASK, SUMMARY, INTENT. Current code has ENTITY, EVENT, FACT, DECISION, PREFERENCE, TASK, SUMMARY, NOTE. Key differences: RFC adds INTENT (from RFC-0013), does not have ENTITY as a MemoryObject type (Entity is a separate model), does not have EVENT or NOTE as memory object types.

5. **Entity model:** RFC-0001 defines Entity as a separate first-class model (not a MemoryObject) with `type` (PERSON/PLACE/ORGANISATION/PROJECT/CONCEPT/TOOL/OTHER), `canonical_name`, `aliases`, `namespace_id`, `created_at`, `last_seen_at`. Current code uses MemoryNode with node_type=ENTITY. This is an architectural difference.

6. **Edge types:** RFC-0001 defines DERIVED_FROM, SUPERSEDES, CONTRADICTS, SUPPORTS, RELATES_TO, ABOUT_ENTITY. Current code has RELATES_TO, SUPERSEDES, DERIVED_FROM, MENTIONS, PART_OF, CAUSED_BY, SUPPORTS, CONTRADICTS, HAS_FACT. Some overlap, some differences. RFC's ABOUT_ENTITY is new. Current MENTIONS, PART_OF, CAUSED_BY, HAS_FACT are not in RFC.

7. **Edge fields:** RFC-0001 requires `created_by` on edges. Current MemoryEdge has `provenance_event_id` but no `created_by`.

8. **Memory Bundle:** RFC-0001 defines the MemoryBundle output structure. Not currently in requirements at all.

9. **Actor types:** RFC-0001 introduces ActorType (HUMAN_USER, SYSTEM_PROCESS, AI_AGENT, EXTERNAL_CONNECTOR). Not in current code.

10. **Source types:** RFC-0001 introduces SourceType (USER_STATED, USER_DEMONSTRATED, SYSTEM_INFERRED, EXTERNAL_DOCUMENT, TOOL_OUTPUT, IMPORTED). Not in current code.

**Triage:**
- Event model field gaps: **P1** (affects Phase 3 retrieval pipeline design, but not blocking current functionality)
- MemoryObject field gaps: **P1** (epistemic_type, source_type needed before Phase 3)
- Lifecycle state naming: **FLAG FOR MANUAL REVIEW** (contradicts STATE.md decisions)
- Entity as separate model: **P2** (architectural refactor, defer to natural point)
- Missing ActorType/SourceType enums: **P1** (needed before Phase 3)

### RFC-0002: Event Store and Append-Only Log
**Type:** Tier 1 -- Storage and Integrity
**Impact on REQUIREMENTS.md:** Defines the full event store schema, operation log, derived state tables, portability artifact format.
**Key deltas with built code:**

1. **Operation log:** RFC-0002 defines a full `operations` table with 20+ operation types (ASSERT, DEPRECATE, SUPERSEDE, ARCHIVE, TOMBSTONE, EPISTEMIC_TRANSITION, DECAY_APPLIED, REINFORCE, RELATE, ENTITY_CREATE, ENTITY_MERGE, etc.). Current code has no operation log at all. This is a major architectural gap but expected -- operation logging was planned for Phase 5 (Self-Organization).

2. **Tamper-evident chain:** RFC-0002 requires `sequence_num` and `prev_hash` fields on events for tamper-evident chaining within streams. Current EventStore has neither.

3. **Derived state tables:** RFC-0002 defines `memory_objects`, `edges`, `entities` tables as derived state. Current code uses DuckDB for events and graph nodes/edges but doesn't frame them as "derived state rebuildable from event log."

4. **Portability artifact:** RFC-0002 defines `memory_pack/` with Parquet files, not DuckDB files. Current REQUIREMENTS.md TRST-01 references "memory_pack/ directory with manifest.json" but doesn't specify Parquet format. This is a delta in artifact format.

5. **Compaction policy:** RFC-0002 specifies when compaction is and isn't allowed. Not in current requirements.

6. **Policy versioning:** RFC-0002 requires `policy_version` on every operation. Not in current code.

7. **Cold start behavior:** RFC-0002 specifies empty results, normal operation, no pre-seeded data. Current code generally does this.

**Triage:**
- Operation log: **P2** (needed for Phase 5, not for current work)
- Tamper-evident chain: **P2** (security hardening, Phase 6-7)
- Portability artifact format: **P2** (Phase 6)
- Compaction policy: **P2** (Phase 5)
- Policy versioning: **P1** (foundation for Phase 5, should be designed into Phase 3)

### RFC-0003: Epistemic State Model
**Type:** Tier 1 -- Storage and Integrity
**Impact on REQUIREMENTS.md:** Defines 7 epistemic types, confidence defaults matrix, epistemic transitions, contradiction modeling, retrieval behavior by epistemic type. This is entirely new -- no equivalent in current REQUIREMENTS.md.
**Key deltas with built code:**

1. **Epistemic types:** 7 types (OBSERVED, ASSERTED, INFERRED, HYPOTHETICAL, CONDITIONAL, DEPRECATED, UNVERIFIED). Current code has no epistemic type field at all.

2. **Default confidence matrix:** Initial confidence values by (epistemic_type, source_type) pair. Current code uses flat 0.5 default confidence.

3. **Conditional claims:** CONDITIONAL type with condition_state (UNKNOWN/TRUE/FALSE/EXPIRED). Not in current code.

4. **Epistemic transitions:** Formal permitted/forbidden transition rules. Not in current code.

5. **Contradiction modeling:** Preserves both objects, creates CONTRADICTION_NOTED operation, creates CONTRADICTS edge. Current code has CONTRADICTS edge type but no formal contradiction modeling procedure.

6. **Retrieval behavior by epistemic type:** DEFAULT vs EXPLICIT retrieval modes with per-type inclusion rules. Not in current code (Phase 3).

7. **Provenance transparency:** Every object must answer 5 questions from logs alone. Partially supported via evidence_refs.

**Triage:**
- Epistemic type field: **P1** (needed before Phase 3 retrieval)
- Confidence matrix: **P1** (should inform Phase 3 scoring)
- Conditional claims: **P2** (advanced feature)
- Epistemic transitions: **P2** (Phase 5 organizer)
- Contradiction modeling: **P1** (partially exists, needs formalization)
- Retrieval behavior by type: **P2** (Phase 3 will implement this)

### RFC-0004: Namespace and Scope Isolation
**Type:** Tier 1 -- Storage and Integrity
**Impact on REQUIREMENTS.md:** Defines full namespace model with 6 types, access policy, retention policy, retrieval isolation, cross-namespace references. Current code has basic 3-value Scope enum (PERSONAL/PROJECT/ORG).
**Key deltas with built code:**

1. **Namespace types:** RFC defines 6 types (PERSONAL, PROJECT, ORGANISATION, AGENT, SYSTEM, SANDBOX). Current code has 3 Scope values (PERSONAL, PROJECT, ORG).

2. **Namespace as first-class entity:** RFC defines Namespace as a full object with `id`, `name`, `type`, `parent_id`, `access_policy`, `retention_policy`, `decay_policy_ref`. Current code uses Scope as a simple enum field on MemoryObject.

3. **Access policy:** RFC defines granular permissions (READ, WRITE, ASSERT, DEPRECATE, ADMIN) per actor per namespace. Not in current code.

4. **Retention policy:** Per-namespace TTL, max event count, expiry actions. Not in current code.

5. **Retrieval isolation:** Vector search must apply namespace filters before returning candidates, not after. Current code uses post-filter for user_id scoping (noted as Phase 1 decision in STATE.md).

6. **Cross-namespace references:** Must be logged as operations. Not in current code.

7. **Namespace hierarchy:** Tree structure with no automatic permission inheritance. Not in current code.

**Triage:**
- Expanded namespace types: **P1** (affects Phase 3 retrieval design)
- Namespace as first-class entity: **P2** (architectural change, can defer)
- Access policy: **P2** (Phase 4+ concern)
- Retention policy: **P2** (Phase 5)
- Retrieval isolation: **P1** (Phase 3 critical requirement, but current post-filter is accepted via STATE.md decision)
- Cross-namespace references: **P2** (advanced feature)

### RFC-0005: Hybrid Retrieval Pipeline
**Type:** Tier 2 -- Retrieval
**Impact on REQUIREMENTS.md:** Defines the full 6-stage retrieval pipeline. Maps to Phase 3 requirements (RETR-01 through RETR-06). Current RETR requirements are high-level; RFC-0005 provides detailed implementation spec.
**Key new concepts:**
- QueryAnalysis structure with intent classification (FACTUAL_LOOKUP, PREFERENCE_CHECK, TASK_STATUS, HISTORICAL, CONTEXTUAL)
- 4 parallel candidate generation paths (graph, vector, lexical, pinned/active)
- path_count multi-path membership signal
- 8-input composite scoring formula with specific default weights
- Embedding version mismatch detection
- Retrieval logging as operations
- Determinism requirements for retrieval

**Triage:** All **P2** -- these are Phase 3 implementation details. Requirements should be added now for Phase 3 planning, but no code delta in built phases.

### RFC-0006: Retrieval Cost and Context Efficiency
**Type:** Tier 2 -- Retrieval
**Impact on REQUIREMENTS.md:** Defines STR (Signal-to-Token Ratio), token cost estimation, 4 representation formats, context packing algorithm, bundle structure. Entirely new -- supplements RETR-06 with detailed implementation.
**Key new concepts:**
- STR metric (composite_score / token_cost)
- Token cost estimation methods (exact, character-based, pre-computed)
- 5 representation levels (REFERENCE, KEY_VALUE, STRUCTURED, PROSE, FULL)
- 3-priority greedy bin-packing algorithm
- Budget overflow degradation rules
- STR-based quality metrics

**Triage:** All **P2** -- Phase 3 implementation. Add requirements now.

### RFC-0007: Decay and Forgetting Model
**Type:** Tier 3 -- Lifecycle
**Impact on REQUIREMENTS.md:** Defines exponential decay function, 5 decay profiles, confidence decay, reinforcement decay, lifecycle transition thresholds, suppression, organizer scheduling, tombstones, decay policy versioning.
**Key new concepts for requirements:**

**Triage:** All **P2** -- Phase 5 implementation. Add requirements now.

### RFC-0008: Confidence Evolution and Reinforcement
**Type:** Tier 3 -- Lifecycle
**Impact on REQUIREMENTS.md:** Defines asymptotic reinforcement, multiplicative penalty, signal types and weights, agent independence problem, saturation controls, epistemic type constraints on reinforcement.
**Key new concepts:**
- Reinforcement formula: `confidence_new = confidence_old + gamma * (1.0 - confidence_old)`
- Penalty formula: `confidence_new = confidence_old * (1.0 - delta)`
- 4 positive signals (REFERENCED, CONFIRMED, TASK_COMPLETED_WITH, CORROBORATED_BY_EXTERNAL)
- 4 negative signals (CORRECTED, CONTRADICTED_BY_OBSERVATION, TASK_FAILED_WITH, IGNORED_REPEATEDLY)
- Agent independence check for cross-agent reinforcement
- Per-session reinforcement cap (0.40)
- Maximum confidence ceiling (0.97)
- Epistemic type confidence ceilings

**Triage:** All **P2** -- Phase 5 implementation. Add requirements now.

## Cross-Cutting Themes

These patterns appear across multiple RFCs and should be captured as overarching requirements:

### Theme 1: Policy Versioning (RFC-0002, 0007, 0008, 0009, 0014)
Every operation resulting from a policy decision must record which policy version was applied. This enables exact replay and audit. Touches: decay, confidence, extraction, retrieval weights.

### Theme 2: `[HYPOTHESIS]` Markers (RFC-0000, 0003, 0005, 0006, 0007, 0008, 0009, 0010)
Many default parameter values are explicitly marked as hypotheses requiring experimental validation. Implementation should use these defaults but expose them as configurable and track them for future calibration.

### Theme 3: Operation Logging (RFC-0002, 0003, 0005, 0007, 0008, 0009)
A comprehensive operation log captures every state change. This is the audit trail that makes the system deterministically rebuildable. Current code has no operation log -- this is a Phase 5+ feature.

### Theme 4: Epistemic Typing (RFC-0001, 0003, 0005, 0008)
Every memory object carries an epistemic type that governs retrieval behavior, confidence evolution, and decay. This cross-cuts storage, retrieval, and lifecycle. It should be added to models before Phase 3.

### Theme 5: Namespace vs Scope (RFC-0004)
The Revised RFCs use "namespace" as a first-class concept with policies, hierarchy, and access control. Current code uses "scope" as a simple enum. The full namespace model is a significant expansion but the basic scope isolation already works for v1 use cases.

## Open Questions

1. **Lifecycle State Naming**
   - What we know: RFC says ACTIVE/DEPRECATED, code uses TENTATIVE/STABLE/SUPERSEDED/ARCHIVED
   - What's unclear: Whether to rename to match RFC or keep current naming and document mapping
   - Recommendation: Flag for manual review per user instruction. Document the mapping in requirements either way.

2. **Entity as Separate Model vs NodeType**
   - What we know: RFC-0001 defines Entity as a separate first-class model distinct from MemoryObject. Current code uses NodeType.ENTITY in the same MemoryNode model.
   - What's unclear: When to make this architectural change (before Phase 3 or later)
   - Recommendation: Add as P1 requirement, address before Phase 3 if it affects retrieval pipeline design.

3. **Scope of "New Requirements"**
   - What we know: User wants complete requirements for all RFCs including future-phase ones
   - What's unclear: How detailed to make Tier 4 requirements (brief summaries vs full conformance lists)
   - Recommendation: Brief summary requirements for Tier 4, detailed requirements for Tier 0-2.

4. **Handling `[HYPOTHESIS]` Values in Requirements**
   - What we know: Many RFC default values are marked as hypotheses
   - What's unclear: Whether requirements should reference specific default values or just say "configurable"
   - Recommendation: Requirements should say "configurable with default of X (RFC-XXXX HYPOTHESIS)" to capture both the configurability requirement and the starting point.
