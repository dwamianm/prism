<decisions>
## Implementation Decisions

### RFC Scope & Priority
- Review the full Revised RFC suite (RFC-0000 through RFC-0014), not just the minimum 0001–0004
- Also cross-reference the two original RFCs (RFC_0001_PRME_Technical_Spec, RFC_0002_RMS_Git_Sync_Profile) to catch anything the Revised suite dropped or changed
- Review in numerical order (0000, 0001, 0002, ... 0014) — no prioritization by build status
- Write complete requirements for all RFCs, including future-phase ones — treat this as a comprehensive requirements pass

### Delta Triage Criteria
- Deltas in already-built features (phases 1–2.2) are automatically higher priority than deltas in unbuilt areas
- When a Revised RFC introduces something entirely new (not in original spec or REQUIREMENTS.md), treat it as a new requirement — add to REQUIREMENTS.md, not just the delta report
- When a Revised RFC contradicts a decision already made in PROJECT.md or STATE.md, flag it for manual review — don't auto-resolve in either direction

### Requirement Traceability
- New requirement domains get new prefixes following existing convention (e.g., DECAY-01, TEMP-01, MAGT-01)
- Each new requirement includes an inline source reference: (RFC-XXXX §N.N)
- When updating existing requirements, modify in-place and add a change note like "(Updated per RFC-0003 §2.1)"
- Requirements from v2/future RFCs go into the existing "v2 Requirements" section of REQUIREMENTS.md

### Claude's Discretion
- Reconciliation report organization (per-RFC vs per-requirement-area vs hybrid)
- Whether to use a single document or split files for output
- Whether to include a summary table in the delta report
- Triage severity scheme (binary fix-now/defer, three-tier, or other)

</decisions>
