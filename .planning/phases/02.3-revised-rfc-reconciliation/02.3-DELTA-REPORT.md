### P1-02: Event model missing `actor_id` and `actor_type` fields

### MR-31: Lifecycle State Naming -- TENTATIVE/STABLE vs ACTIVE/DEPRECATED

**Conflict:**
- **RFC-0001 S8** defines lifecycle states as: ACTIVE, DEPRECATED, SUPERSEDED, ARCHIVED. The RFC collapses TENTATIVE and STABLE into a single ACTIVE state and introduces DEPRECATED (an object determined to be incorrect or outdated).
- **Built code** (`src/prme/types.py`) uses: TENTATIVE, STABLE, SUPERSEDED, ARCHIVED. The TENTATIVE-to-STABLE promotion is a core lifecycle transition in the current implementation (GraphStore.promote() method).
- **STATE.md decision** (Phase 01): "Event model uses frozen=True for immutability; MemoryEdge inherits BaseModel (not MemoryObject)" -- lifecycle states established early.
- **REQUIREMENTS.md STOR-07** is already flagged with MANUAL REVIEW marker.

**RFC rationale for the change:** RFC-0003 (Epistemic State Model) replaces the TENTATIVE/STABLE distinction with epistemic types and confidence evolution. An object's "stability" is expressed through its confidence score and epistemic type, not a separate lifecycle state. The RFC's DEPRECATED state serves a different purpose -- marking objects as definitively incorrect/outdated (irreversible).

**Options:**
1. **Align with RFC:** Rename to ACTIVE/DEPRECATED/SUPERSEDED/ARCHIVED. Remove `promote()` method. Express stability through confidence thresholds. Breaking change across codebase.
2. **Keep current naming:** Document the mapping (TENTATIVE+STABLE = ACTIVE, add DEPRECATED as new state). Keep `promote()` as a confidence threshold transition. Less disruptive.
3. **Hybrid:** Keep TENTATIVE/STABLE but add DEPRECATED. Four states become five. Promotion semantics preserved, deprecation concept added.

**Impact on Phase 3:** Phase 3 retrieval needs to filter by lifecycle state. The choice of naming/semantics affects retrieval filter design. This should be resolved before Phase 3 planning begins.

**Decision required:** User must choose between options 1, 2, or 3.

### MR-32: Post-filter vs Pre-filter for Namespace Isolation

**Conflict:**
- **RFC-0004 S6** requires namespace filters to be applied BEFORE returning vector search candidates. Post-hoc filtering is explicitly stated as insufficient for namespace isolation.
- **Built code** uses post-filter strategy for user_id scoping in USearch vector index. The `filtered_search` API was not available in USearch v2.23.0.
- **STATE.md decision** (Phase 01): "Used post-filter strategy for USearch user_id scoping (filtered_search not available in v2.23.0)" -- this was a deliberate technical decision.

**RFC rationale:** Pre-filtering ensures that the k candidates returned by vector search are all from the correct namespace. Post-filtering can return fewer than k relevant results if many candidates are from other namespaces, degrading retrieval quality.

**Options:**
1. **Switch to pre-filter:** Investigate USearch API updates or alternative vector search libraries that support pre-filtering. May require library upgrade or replacement.
2. **Keep post-filter with mitigation:** Over-fetch candidates (e.g., request 3x k) and post-filter to k. Documents the deviation from RFC. Simpler but not spec-compliant.
3. **Defer to Phase 3:** The retrieval pipeline (Phase 3) will redesign the vector search interface. Address pre-filter requirement as part of that design.

**Impact on Phase 3:** This directly affects Phase 3 retrieval pipeline architecture. Should be evaluated during Phase 3 planning, not necessarily resolved now.

**Decision required:** User should decide before Phase 3 planning. Option 3 (defer to Phase 3 design) is recommended.
