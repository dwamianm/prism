# Phase 3.3: Contradiction Modeling - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

When conflicting assertions are detected during ingestion, the system preserves both objects, creates a CONTRADICTS edge, logs a CONTRADICTION_NOTED operation, and surfaces conflicts at retrieval time. Includes a basic resolution method. Does NOT include UI/API for resolution, scheduled contradiction scanning, or automated resolution.

</domain>

<decisions>
## Implementation Decisions

### Contradiction vs Supersedence Boundary
- Use **recency signals via LLM extraction** to distinguish contradictions from supersedence
- Add a `temporal_intent` field to the extraction schema so the LLM classifies whether a conflicting fact is an update to prior state or a new assertion
- If temporal_intent indicates "update" (e.g., "now works at", "moved to") → SUPERSEDES edge (old replaced)
- If temporal_intent indicates "assertion" with no temporal signal → CONTRADICTS edge (both preserved)
- **Default when ambiguous:** supersedence (matches current behavior — assumes newer info is better)

### Object Lifecycle After Contradiction
- Introduce a new lifecycle state: **CONTESTED**
- When a true contradiction is detected, both the old and new objects transition to CONTESTED
- CONTESTED objects remain in the system but are flagged as having an unresolved conflict
- **Resolution flow:** Winner → STABLE, loser → DEPRECATED
- Include a basic `resolve_contradiction()` method (no UI/API, just core logic) that accepts a winner_id and performs state transitions

### Detection Scope
- Contradiction detection applies to **all node types**, not just Facts
- Preferences ("prefers dark mode" vs "prefers light mode"), Decisions ("decided React" vs "decided Vue"), and other types are all candidates
- Same-session contradictions ARE detected (user may change mind mid-conversation)

### Retrieval Impact
- Phase 3.3 includes **end-to-end retrieval changes** — contradictions are surfaced to the consuming LLM
- CONTESTED objects are **excluded from the "stable facts" section** of context bundles
- Contradicting counterparts are NOT automatically pulled into results — only included if independently relevant to the query
- Each CONTESTED result carries conflict metadata (conflict_flag, contradicting object reference)

### Claude's Discretion
- Temporal_intent field design (binary, ternary, or richer classification)
- Contradiction matching strategy for non-Fact types (domain matching, LLM-assessed, or hybrid)
- Contradiction detection directionality (bidirectional vs ingestion-triggered)
- CONTESTED object scoring approach (penalty vs no penalty in retrieval ranking)
- Retrieval presentation of contradiction pairs (grouped vs inline with flags)

</decisions>

<specifics>
## Specific Ideas

- Resolution transitions are clean: winner → STABLE (confirmed correct), loser → DEPRECATED (explicitly wrong)
- The system should detect contradictions even within a single conversation session
- Contradicting counterparts should NOT be injected into retrieval results just because they contradict a matched object — only surface them if they independently score above threshold

</specifics>

<deferred>
## Deferred Ideas

- Automated contradiction resolution via self-organization (Phase 5)
- UI/API for manual contradiction resolution (Phase 4 or later)
- Scheduled scanning for contradictions across existing objects (Phase 5)

</deferred>
