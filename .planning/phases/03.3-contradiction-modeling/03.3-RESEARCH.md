# Phase 3.3: Contradiction Modeling - Research

**Researched:** 2026-02-21
**Domain:** Contradiction detection at ingestion time, CONTESTED lifecycle state, CONTRADICTS edge creation, operation logging, retrieval impact
**Confidence:** HIGH

## Summary

Phase 3.3 closes EPIS-04 by splitting the current `SupersedenceDetector` into a branching decision point: when conflicting assertions are detected, the system must decide whether this is a supersedence (update) or a true contradiction (conflicting claims). The user has locked the decision that this distinction is driven by a `temporal_intent` field on the extraction schema -- the LLM classifies whether a conflicting fact is an update or a new assertion. Supersedence continues to work as today; true contradictions create CONTRADICTS edges, transition both objects to CONTESTED, and log a CONTRADICTION_NOTED operation.

The codebase is well-positioned for this work. The `EdgeType.CONTRADICTS` enum member already exists in `types.py`. The `MemoryEdge` model, the `GraphWriter` protocol, and the `WriteQueueGraphWriter` implementation provide the write infrastructure. The `SupersedenceDetector` in `ingestion/supersedence.py` contains the exact detection logic (subject+predicate match, differing object value) that needs to be extended with the temporal_intent branching. The operations table schema (`operations` in `schema.py`) already supports structured JSON payloads and is used by the retrieval pipeline for RETRIEVAL_REQUEST logging. The retrieval pipeline has all the necessary infrastructure for filtering and metadata annotation -- `filtering.py` for epistemic filtering, `packing.py` for section classification, and `RetrievalCandidate`/`RetrievalResponse` models for metadata.

**Primary recommendation:** Extend in dependency order: (1) add `temporal_intent` to extraction schema and prompt, (2) add CONTESTED lifecycle state and wire transitions, (3) modify SupersedenceDetector to branch on temporal_intent, (4) create CONTRADICTS edges and log CONTRADICTION_NOTED operations, (5) add `resolve_contradiction()` method, (6) wire retrieval changes (exclude CONTESTED from stable_facts, add conflict metadata to results).

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Use **recency signals via LLM extraction** to distinguish contradictions from supersedence
- Add a `temporal_intent` field to the extraction schema so the LLM classifies whether a conflicting fact is an update to prior state or a new assertion
- If temporal_intent indicates "update" (e.g., "now works at", "moved to") -> SUPERSEDES edge (old replaced)
- If temporal_intent indicates "assertion" with no temporal signal -> CONTRADICTS edge (both preserved)
- **Default when ambiguous:** supersedence (matches current behavior -- assumes newer info is better)
- Introduce a new lifecycle state: **CONTESTED**
- When a true contradiction is detected, both the old and new objects transition to CONTESTED
- CONTESTED objects remain in the system but are flagged as having an unresolved conflict
- **Resolution flow:** Winner -> STABLE, loser -> DEPRECATED
- Include a basic `resolve_contradiction()` method (no UI/API, just core logic) that accepts a winner_id and performs state transitions
- Contradiction detection applies to **all node types**, not just Facts
- Preferences ("prefers dark mode" vs "prefers light mode"), Decisions ("decided React" vs "decided Vue"), and other types are all candidates
- Same-session contradictions ARE detected (user may change mind mid-conversation)
- Phase 3.3 includes **end-to-end retrieval changes** -- contradictions are surfaced to the consuming LLM
- CONTESTED objects are **excluded from the "stable facts" section** of context bundles
- Contradicting counterparts are NOT automatically pulled into results -- only included if independently relevant to the query
- Each CONTESTED result carries conflict metadata (conflict_flag, contradicting object reference)

### Pattern 1: Temporal Intent Branching in SupersedenceDetector

**What:** When a predicate match with differing object value is found, check `temporal_intent` to decide SUPERSEDES vs CONTRADICTS.
**When to use:** Every time the existing contradiction detection logic in `detect_and_supersede` fires.

```python

if _predicates_match(predicate, existing_predicate):
    if existing_object != object_value:
        if temporal_intent == "update":
            # Current behavior: supersede
            await self._graph_writer.supersede(...)
        else:
            # New behavior: contradict
            await self._graph_writer.contradict(
                node_a_id=target_id,
                node_b_id=new_fact_node_id,
                evidence_id=evidence_event_id,
            )
```

### Pattern 2: CONTESTED Lifecycle State Machine Extension

**What:** Add CONTESTED as a valid state with specific transition rules.
**When to use:** When both objects in a contradiction need state transitions.

```python
# Extended ALLOWED_TRANSITIONS in types.py
LifecycleState.TENTATIVE: {STABLE, SUPERSEDED, ARCHIVED, CONTESTED},
LifecycleState.STABLE: {SUPERSEDED, ARCHIVED, CONTESTED},
LifecycleState.CONTESTED: {STABLE, DEPRECATED, ARCHIVED},
# DEPRECATED is a new terminal-ish state for contradiction losers
```

Note: The existing lifecycle has SUPERSEDED but no DEPRECATED. Per the user decision, resolution transitions are: winner -> STABLE, loser -> DEPRECATED. Since DEPRECATED does not currently exist in the LifecycleState enum, there are two options:
1. **Reuse SUPERSEDED for loser** -- semantically close ("this was replaced by better information")
2. **Add DEPRECATED to LifecycleState** -- matches RFC-0003 terminology exactly

Recommendation: **Add DEPRECATED** to LifecycleState. The RFC is explicit about DEPRECATED as a distinct state (RFC-0003 S6: "DEPRECATED -- previously valid but now determined to be incorrect, outdated, or superseded"). SUPERSEDED means "replaced by newer info" while DEPRECATED means "determined to be wrong." These are semantically different. The existing SUPERSEDED remains for supersedence chains; DEPRECATED is for contradiction losers.

### Anti-Patterns to Avoid
- **Auto-resolving contradictions:** RFC-0003 S7 says "NOT automatically deprecate either object." The system MUST preserve both and wait for explicit resolution.
- **Injecting contradicting counterparts into results:** Per locked decision, counterparts are NOT automatically pulled in -- only surface if independently relevant to the query.
- **Bidirectional edge creation:** Only ONE CONTRADICTS edge is needed (from the newer node to the older). The retrieval pipeline can look up both directions.
- **Modifying existing supersedence behavior:** The current SUPERSEDES path must continue working identically for temporal_intent="update" cases. The branching only adds a new path for temporal_intent="assertion".

## Common Pitfalls

### Pitfall 1: Breaking Existing Supersedence
**What goes wrong:** Modifying SupersedenceDetector changes behavior for ALL detected conflicts, breaking existing supersedence chains.
**Why it happens:** The new temporal_intent branching is inserted into the existing detect_and_supersede method.
**How to avoid:** Default temporal_intent to "update" (locked decision: "Default when ambiguous: supersedence"). If temporal_intent is None or missing (pre-migration facts, old extraction results), fall through to existing supersedence behavior. Only branch to CONTRADICTS when temporal_intent explicitly equals "assertion".
**Warning signs:** Existing ingestion tests start failing; facts that should be superseded are instead marked as contradictions.

### Pitfall 2: CONTESTED State Transition Inconsistency
**What goes wrong:** A node in CONTESTED state can't be promoted, superseded, or archived because ALLOWED_TRANSITIONS doesn't include those paths.
**Why it happens:** Incomplete transition table update when adding CONTESTED.
**How to avoid:** Map all valid transitions from CONTESTED: CONTESTED -> STABLE (winner resolution), CONTESTED -> DEPRECATED (loser resolution), CONTESTED -> ARCHIVED (archival). Add DEPRECATED -> ARCHIVED as well for completeness.
**Warning signs:** ValueError exceptions when resolving contradictions or running organizer jobs on CONTESTED nodes.

### Pitfall 3: Query_nodes Default Filter Excludes CONTESTED
**What goes wrong:** CONTESTED nodes become invisible in retrieval because `query_nodes` defaults to `lifecycle_states=[TENTATIVE, STABLE]`.
**Why it happens:** New lifecycle state not included in default filter.
**How to avoid:** Add CONTESTED to the default active states list in DuckPGQGraphStore._query_nodes_sync. CONTESTED nodes ARE active (they have unresolved conflicts but are still valid candidates for retrieval). The retrieval pipeline handles presentation separately.
**Warning signs:** CONTESTED nodes never appear in retrieval results even when independently relevant.

### Pitfall 4: Retrieval Conflict Metadata Lookup Performance
**What goes wrong:** For each CONTESTED candidate, looking up the CONTRADICTS edge to find the counterpart adds N extra queries.
**Why it happens:** Naive per-candidate edge lookup in the retrieval pipeline.
**How to avoid:** Batch-query CONTRADICTS edges for all CONTESTED candidates in a single pass after candidate merging. Use `get_edges(source_id=..., edge_type=EdgeType.CONTRADICTS)` + `get_edges(target_id=..., edge_type=EdgeType.CONTRADICTS)` once per batch, not once per candidate.
**Warning signs:** Retrieval latency increases linearly with number of CONTESTED nodes in results.

### Pitfall 5: Extraction Schema Backward Compatibility
**What goes wrong:** Adding temporal_intent as a required field breaks existing extraction results and tests.
**Why it happens:** ExtractedFact is used by instructor with max_retries -- the LLM must produce the field.
**How to avoid:** Make temporal_intent an Optional field with a default of None. When None, use the existing behavior (treat as "update" per the default-to-supersedence rule). This preserves backward compatibility and handles cases where the LLM fails to classify.
**Warning signs:** Instructor retries exhaust on old messages where temporal intent is genuinely ambiguous.

### Pitfall 6: DuckDB UPDATE-as-DELETE+INSERT with CONTRADICTS Edges
**What goes wrong:** When transitioning a node to CONTESTED, the DuckDB UPDATE internally does DELETE+INSERT. If edges reference that node_id, the FK-free design handles this fine, but the edge's provenance_event_id might become stale.
**Why it happens:** Same issue documented for Phase 01 ("Removed FK constraints from edges table -- DuckDB treats UPDATE as DELETE+INSERT").
**How to avoid:** This is already handled by the FK-free design. No additional action needed. The CONTRADICTS edge references node IDs which remain stable through the UPDATE.
**Warning signs:** None expected -- this is already mitigated by existing design.

### Example 3: GraphStore.contradict() Method

```python
# In storage/duckpgq_graph.py - DuckPGQGraphStore
def _contradict_sync(
    self,
    node_a_id: str,
    node_b_id: str,
    evidence_id: str | None,
) -> None:
    """Mark two nodes as contradicting each other (sync)."""
    # Validate both nodes exist and are active
    for node_id, label in [(node_a_id, "Node A"), (node_b_id, "Node B")]:
        row = self._conn.execute(
            "SELECT lifecycle_state, user_id FROM nodes WHERE id = ?",
            [node_id],
        ).fetchone()
        if row is None:
            raise ValueError(f"{label} {node_id} not found")
        state = LifecycleState(row[0])
        if state not in (LifecycleState.TENTATIVE, LifecycleState.STABLE):
            raise ValueError(
                f"Cannot contest: {label} is {state.value}, "
                "only Tentative or Stable nodes can be contested"
            )

    # Transition both to CONTESTED
    for node_id in [node_a_id, node_b_id]:
        self._conn.execute(
            "UPDATE nodes SET lifecycle_state = ?, updated_at = current_timestamp WHERE id = ?",
            ["contested", node_id],
        )

    # Create CONTRADICTS edge: node_b (newer) -> node_a (older)
    provenance_uuid = UUID(evidence_id) if evidence_id else None
    edge = MemoryEdge(
        source_id=UUID(node_b_id),
        target_id=UUID(node_a_id),
        edge_type=EdgeType.CONTRADICTS,
        user_id=row[1],  # user_id from node validation
        confidence=1.0,
        provenance_event_id=provenance_uuid,
    )
    self._create_edge_sync(edge)

    # Log CONTRADICTION_NOTED operation
    import json, uuid as uuid_mod
    op_id = str(uuid_mod.uuid4())
    payload = json.dumps({
        "node_a_id": node_a_id,
        "node_b_id": node_b_id,
        "evidence_event_id": evidence_id,
    })
    self._conn.execute(
        "INSERT INTO operations (id, op_type, target_id, payload, actor_id, created_at) "
        "VALUES (?, ?, ?, ?, ?, now())",
        [op_id, "CONTRADICTION_NOTED", node_a_id, payload, "system"],
    )
```

### Example 4: Retrieval Conflict Metadata Annotation

```python
# In retrieval/pipeline.py - after scoring, before packing
# Batch-annotate CONTESTED candidates with conflict metadata
contested_ids = [
    str(c.node.id) for c in scored
    if c.node.lifecycle_state == LifecycleState.CONTESTED
]
if contested_ids:
    # Batch lookup CONTRADICTS edges
    for cid in contested_ids:
        edges = await self._graph_store.get_edges(
            source_id=cid, edge_type=EdgeType.CONTRADICTS
        )
        edges += await self._graph_store.get_edges(
            target_id=cid, edge_type=EdgeType.CONTRADICTS
        )
        if edges:
            counterpart_id = (
                str(edges[0].target_id) if str(edges[0].source_id) == cid
                else str(edges[0].source_id)
            )
            # Find the candidate and annotate
            for c in scored:
                if str(c.node.id) == cid:
                    c.conflict_flag = True
                    c.contradicts_id = UUID(counterpart_id)
```

## Discretion Recommendations

For areas marked as Claude's discretion in CONTEXT.md:

### 1. temporal_intent Field Design
**Recommendation: Ternary** -- "update", "assertion", None (null).
- Binary forces incorrect classification on ambiguous cases
- Ternary maps cleanly to the locked default (None -> supersedence)
- Aligns with P6 (Empirical Humility) and [BEST-EFFORT] extraction principle

### 2. Contradiction Matching Strategy for Non-Fact Types
**Recommendation: Same predicate matching for all types.** Preferences and Decisions already use the subject-predicate-object triple structure in ExtractedFact (via fact_type="preference"/"decision"). The existing `_predicates_match()` logic works unchanged. For types without clear predicates, no matching is attempted (consistent with current behavior where only subject+predicate contradictions are detected).

### 3. Contradiction Detection Directionality
**Recommendation: Ingestion-triggered only (unidirectional).** Detection runs when a new fact is ingested, checking against existing facts. This is the current pattern for supersedence. Bidirectional scanning (checking all pairs at rest) is explicitly deferred to Phase 5.

### 4. CONTESTED Object Scoring Approach
**Recommendation: No scoring penalty.** CONTESTED objects should score normally based on their content relevance. The conflict_flag metadata tells the consuming LLM about the unresolved conflict. Applying a scoring penalty would bury relevant but contested information, which violates P3 (Epistemic Honesty) -- the system should present conflicts transparently, not hide them.

### 5. Retrieval Presentation of Contradiction Pairs
**Recommendation: Inline with flags.** Each CONTESTED candidate carries `conflict_flag=True` and `contradicts_id=<uuid>`. The consuming LLM can look up the counterpart if needed. This is simpler than grouping and aligns with the locked decision that "contradicting counterparts are NOT automatically pulled into results."

## Open Questions

1. **DEPRECATED vs SUPERSEDED for contradiction losers**
   - What we know: User decision says "loser -> DEPRECATED". RFC-0003 uses DEPRECATED terminology. Current codebase has SUPERSEDED but not DEPRECATED.
   - What's unclear: Whether adding DEPRECATED creates confusion with the existing SUPERSEDED state.
   - Recommendation: Add DEPRECATED as distinct from SUPERSEDED. They are semantically different: SUPERSEDED = "replaced by newer info (both might have been correct at different times)" vs DEPRECATED = "determined to be wrong." This aligns with RFC-0003 S3 and S6. The two states can coexist cleanly. DEPRECATED objects should also be excluded from DEFAULT retrieval mode (add to `DEFAULT_EXCLUDED_EPISTEMIC` or handle via lifecycle filtering).

2. **CONTESTED inclusion in DEFAULT retrieval mode**
   - What we know: CONTESTED objects should be visible (they're not wrong, just disputed). But they should NOT be in "stable facts" section.
   - What's unclear: Whether CONTESTED should be in DEFAULT_EXCLUDED_EPISTEMIC or handled separately.
   - Recommendation: CONTESTED is a lifecycle state, not an epistemic type. Handle it in the packing/section-classification stage, not in epistemic filtering. Include CONTESTED candidates in retrieval results but classify them into a different section (e.g., "contested_claims" instead of "stable_facts"). This keeps epistemic filtering clean and lifecycle filtering separate.

3. **Operations table actor_id for system-detected contradictions**
   - What we know: The operations table has an `actor_id` column. Contradiction detection is system-initiated.
   - What's unclear: What actor_id to use for system-initiated contradiction detection.
   - Recommendation: Use "system" as actor_id for ingestion-triggered contradictions. For user-initiated resolution, use the resolver's user_id.
