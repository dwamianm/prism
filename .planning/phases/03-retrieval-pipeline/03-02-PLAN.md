  <name>Task 1: Build query analysis module</name>
  <files>src/prme/retrieval/query_analysis.py</files>
  <action>
    Create `src/prme/retrieval/query_analysis.py` implementing Stage 1 of the retrieval pipeline per RFC-0005 §3.

    **IMPORTANT:** Query analysis MUST NOT make blocking LLM calls (RFC-0005 §3 explicitly prohibits this by default). Use dateparser + pattern matching only.

    **Function: `async def analyze_query(query: str, *, time_from: datetime | None = None, time_to: datetime | None = None, retrieval_mode: RetrievalMode = RetrievalMode.DEFAULT) -> QueryAnalysis`**

    Implementation:

    1. **Intent classification** (simple keyword/pattern matching, NOT an LLM call):
       - ENTITY_LOOKUP: Query starts with "who", "what is", contains proper nouns detected by capitalization heuristic (e.g., consecutive capitalized words not at sentence start)
       - TEMPORAL: Query contains temporal keywords ("when", "last", "before", "after", "recent", "ago", "yesterday", "today") OR dateparser finds temporal expressions
       - RELATIONAL: Query contains relationship keywords ("related to", "connected", "linked", "between", "associated")
       - FACTUAL: Query contains factual keywords ("what", "how", "why", "does", "is it true", "fact")
       - SEMANTIC: Default fallback — any query that doesn't match specific patterns

       Intent should be the FIRST matching pattern, with SEMANTIC as fallback. This is a simple heuristic, not a classifier.

    2. **Entity extraction** (simple heuristic, NOT an LLM call):
       - Extract sequences of capitalized words (2+ consecutive) as potential entity names
       - Extract quoted strings as entity references
       - Return as list[str] — downstream can look these up in the graph store

    3. **Temporal signal extraction:**
       - Use `dateparser.search.search_dates(query, settings={'RETURN_AS_TIMEZONE_AWARE': True, 'TIMEZONE': 'UTC'})` to find temporal expressions
       - For each result: classify as ABSOLUTE (contains digits) or RELATIVE (no digits)
       - Store as list of dicts: `{"type": "ABSOLUTE"|"RELATIVE", "value": str, "resolved": datetime}`
       - If caller provided explicit `time_from`/`time_to`, those override any extracted signals. If not provided and temporal signals found, derive time_from/time_to from the resolved datetimes.

    4. **Construct and return QueryAnalysis** with all extracted data plus a new UUID request_id.

    **Error handling:** If dateparser fails or is unavailable, return empty temporal_signals and log a warning. Never crash on analysis — this is Stage 1, downstream stages must still work.

    Add module-level logger: `logger = logging.getLogger(__name__)`
  </action>

<task type="auto">
  <name>Task 2: Build parallel candidate generation and merging</name>
  <files>src/prme/retrieval/candidates.py</files>
  <action>
    Create `src/prme/retrieval/candidates.py` implementing Stages 2 (candidate generation) and 3 (merging) per RFC-0005 §4-5.

    **Function: `async def generate_candidates(analysis: QueryAnalysis, *, graph_store: GraphStore, vector_index: VectorIndex, lexical_index: LexicalIndex, user_id: str, scope: Scope | None = None, config: PackingConfig = DEFAULT_PACKING_CONFIG) -> tuple[list[RetrievalCandidate], dict[str, int]]`**

    Returns: (merged_candidates, candidate_counts_per_backend)

    Implementation:

    1. **Entity lookup for graph seeding:**
       If `analysis.entities` is non-empty, query `graph_store.query_nodes(user_id=user_id, node_type=NodeType.ENTITY)` and find nodes whose content matches extracted entity names (case-insensitive). These are the seed nodes for graph traversal.

    2. **Parallel candidate generation via asyncio.gather:**
       Run up to 4 async coroutines in parallel:

       a. **Graph path** (if seed nodes found):
          For each seed node, call `graph_store.get_neighborhood(seed_id, max_hops=config.graph_max_hops)`.
          Post-filter to max `config.graph_max_candidates` total (sort by confidence DESC, then created_at DESC, truncate).
          Map hop distance to graph_proximity: 1-hop=1.0, 2-hop=0.7, 3-hop=0.4.
          **Note:** Current get_neighborhood() returns flat list without hop info. Use a helper that runs get_neighborhood at max_hops=1, then max_hops=2 (subtract 1-hop results), then max_hops=3 (subtract 1&2-hop) to determine per-node hop distances. If too complex, approximate: all results from max_hops=1 get 1.0, remaining from max_hops=2 get 0.7, remaining from max_hops=3 get 0.4.

       b. **Vector path:**
          Call `vector_index.search(analysis.query, user_id, k=config.vector_k)`.
          Results are list[dict] with keys: node_id, score, distance.
          The `score` is already cosine similarity in [0,1] — use as semantic_score.
          **Embedding version mismatch:** Wrap in try/except. If the search fails, log warning and set embedding_mismatch flag. Return empty vector candidates.

       c. **Lexical path:**
          Call `lexical_index.search(analysis.query, user_id, limit=config.lexical_k)`.
          Results are list[dict] with keys: node_id, content, score, node_type.
          BM25 scores are unbounded — normalize using per-query min-max: `(score - min) / (max - min)`. If all scores equal, set all to 1.0.

       d. **Pinned/active path:**
          Query `graph_store.query_nodes(user_id=user_id, min_confidence=None)` filtering for nodes with `salience == 1.0` (pinned) OR `node_type == NodeType.TASK` with lifecycle_state in (TENTATIVE, STABLE). These are always-include candidates.

       Use `asyncio.gather(*coros, return_exceptions=True)` to handle individual backend failures gracefully. If a backend raises, log warning and treat as empty result.

    3. **BM25 normalization helper:**
       `def normalize_bm25_scores(results: list[dict]) -> list[dict]` — min-max normalization per research §Code Examples. Adds `normalized_score` key to each result dict.

    4. **Candidate merging:**
       `def merge_candidates(graph_cands, vector_cands, lexical_cands, pinned_cands, graph_store) -> list[RetrievalCandidate]`

       Merge by `node.id`:
       - If a node appears in multiple backends, combine: union of paths, increment path_count, take max of each score component.
       - For vector-only candidates, we only have node_id (not full MemoryNode). These need to be batch-resolved from the graph store. Call `graph_store.get_node(node_id)` for each. If node not found (orphaned vector entry), skip.
       - For lexical-only candidates, similarly resolve from graph store.
       - Graph and pinned candidates already have full MemoryNode objects.

       **Batch resolution strategy:** Collect all node_ids that need resolution, deduplicate, and resolve in a loop (DuckDB is local, no network latency concern). This avoids the anti-pattern of eager full-node fetch during generation — we resolve AFTER generation.

    5. **Return** `(merged_candidates, counts_dict)` where counts_dict = {"GRAPH": N, "VECTOR": N, "LEXICAL": N, "PINNED": N}.

    Add module-level logger. Handle all exceptions gracefully — a failing backend should not crash the entire retrieval.
  </action>
