# Phase 3.1: Epistemic Type & Confidence Matrix - Context

**Gathered:** 2026-02-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Every memory object carries its epistemic type at creation, default confidence is assigned from a (type, source) matrix, and epistemic filtering uses real field values instead of getattr fallback. Closes audit gaps for EPIS-01, EPIS-02, and EPIS-05.

</domain>

<decisions>
## Implementation Decisions

### Confidence matrix defaults
- OBSERVED facts from user input get high initial confidence (0.85–0.90)
- Claude determines all other default values aligned with RFC [HYPOTHESIS] markers and P3 (Epistemic Honesty)
- Matrix stored as Python dataclass with config file override — defaults in code, overridable via config if present
- Claude determines whether to mark individual matrix cells as hypothesis vs validated based on P6 (Empirical Humility) and practical utility

### Source type taxonomy
- source_type persisted as a field on MemoryNode (not just used at ingestion time) — enables provenance auditing
- Claude determines granularity of source types based on what the codebase actually needs to distinguish
- Claude determines whether source_type is a fixed enum or extensible with fallback, based on system evolution needs
- Claude audits existing provenance tracking in the codebase and picks the pragmatic alignment path

### Existing data migration
- Pre-existing MemoryNodes without epistemic_type get backfilled using best-guess heuristics (infer type from node type / content patterns)
- Backfilled nodes are marked as retroactively inferred — a flag or metadata distinguishes them from nodes typed at creation
- Migration runs automatically on startup when untyped nodes are detected
- Existing confidence values are preserved during backfill — only epistemic_type is assigned, confidence is not recalculated from the matrix

### Type assignment logic
- System infers epistemic type by default based on context (node type, source), but caller can override with an explicit type
- LLM extraction prompts include epistemic type classification — the LLM identifies whether extracted content is OBSERVED, ASSERTED, INFERRED, etc.
- Strict validation on LLM-returned epistemic types — reject entire extraction if type is invalid (no silent fallback)
- Creation-time types restricted to: OBSERVED, ASSERTED, INFERRED, HYPOTHETICAL, CONDITIONAL, UNVERIFIED. DEPRECATED is a lifecycle transition only, not assignable at creation.
- Epistemic type assignment logged as an operation in the operation log (EPISTEMIC_TYPE_ASSIGNED) for full audit trail

### Claude's Discretion
- Exact numeric values in the confidence matrix (except OBSERVED from user input: 0.85–0.90)
- Source type taxonomy granularity and specific type names
- Whether source_type is fixed or extensible
- Whether matrix cells carry hypothesis markers
- Alignment strategy between existing provenance tracking and new source_type field

</decisions>

<specifics>
## Specific Ideas

- User wants epistemic honesty prioritized — OBSERVED facts from direct user statements deserve high confidence, everything else should earn its confidence through reinforcement
- Strict validation over graceful degradation for LLM classification — better to reject bad data than silently accept wrong epistemic types
- Migration should be invisible to the user (auto on startup) but transparent in the data (marked as retroactively inferred)
- Existing confidence values are sacred during migration — the system should only add epistemic_type, never downgrade confidence retroactively

</specifics>
