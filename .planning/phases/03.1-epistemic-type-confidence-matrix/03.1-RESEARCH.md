# Phase 3.1: Epistemic Type & Confidence Matrix - Research

**Researched:** 2026-02-20
**Domain:** Pydantic data model evolution, DuckDB schema migration, epistemic classification in LLM extraction pipelines
**Confidence:** HIGH

## Summary

Phase 3.1 closes three audit gaps (EPIS-01, EPIS-02, EPIS-05) by making epistemic type a first-class field on every memory object. The codebase already has `EpistemicType` as an enum in `types.py`, `EPISTEMIC_WEIGHTS` lookup tables, and `DEFAULT_EXCLUDED_EPISTEMIC` sets used by both scoring (`scoring.py`) and filtering (`filtering.py`). However, `MemoryNode` in `models/nodes.py` has no `epistemic_type` field -- the retrieval pipeline uses `getattr(node, "epistemic_type", EpistemicType.ASSERTED)` as a forward-compatible fallback in two locations (`filtering.py:37` and `scoring.py:49`). Tests in `test_retrieval_scoring.py` use `object.__setattr__` to monkey-patch epistemic_type onto nodes because the field does not exist.

The implementation is straightforward: add the field to MemoryNode, add the column to DuckDB, wire it through the serialization layer (`_row_to_node`/`_create_node_sync`), build the confidence matrix as a dataclass, update extraction prompts for LLM epistemic classification, update the ingestion pipeline to assign epistemic type and matrix-derived confidence at creation, build a migration for existing data, and remove the getattr fallbacks.

**Primary recommendation:** This is a model-layer change with cascading updates to storage, ingestion, and retrieval. Execute in dependency order: types/model first, then schema/storage, then confidence matrix, then ingestion integration, then retrieval cleanup, then migration, then tests.

### Pattern 1: Column-Positional Sensitivity in _row_to_node
**What:** DuckPGQ's `_row_to_node` uses positional indexing (`row[0]`, `row[1]`, ..., `row[15]`) to map DuckDB columns to MemoryNode fields. Adding new columns shifts all positions after the insertion point.
**When to use:** Whenever adding columns to the nodes table.
**Critical detail:** The current column order in CREATE TABLE (16 columns) maps directly to the positional indices in `_row_to_node`. New columns (`epistemic_type`, `source_type`) must be added at a known position and ALL positional indices after that point must be updated.

**Current column order (indices 0-15):**
```
0:  id
1:  node_type
2:  user_id
3:  session_id
4:  scope
5:  content
6:  metadata
7:  confidence
8:  salience
9:  lifecycle_state
10: valid_from
11: valid_to
12: superseded_by
13: evidence_refs
14: created_at
15: updated_at
```

**Recommendation:** Add `epistemic_type` and `source_type` as columns 16 and 17 (at the end of the table) via ALTER TABLE. This avoids shifting existing indices. For new databases, add them to the CREATE TABLE in the same position. The `_row_to_node` method adds `row[16]` and `row[17]` handling. All `SELECT *` queries automatically include the new columns.

### Pattern 3: ConfidenceMatrix as Frozen Dataclass
**What:** The (epistemic_type, source_type) confidence matrix is a frozen Pydantic model with dict-based lookup.
**When to use:** At node creation time, when default confidence needs to be assigned.

```python
from pydantic import BaseModel, ConfigDict

class ConfidenceMatrix(BaseModel):
    """Default confidence values by (epistemic_type, source_type).

    All values are [HYPOTHESIS] -- tunable per deployment.
    """
    model_config = ConfigDict(frozen=True)

    matrix: dict[tuple[str, str], float] = {
        # (epistemic_type, source_type) -> default confidence
        ("observed", "user_stated"): 0.90,
        ("observed", "user_demonstrated"): 0.85,
        # ... etc
    }

    def lookup(
        self,
        epistemic_type: EpistemicType,
        source_type: SourceType,
    ) -> float | None:
        """Return default confidence, or None for invalid combinations."""
        return self.matrix.get((epistemic_type.value, source_type.value))
```

### Pattern 4: Strict Validation on LLM Extraction
**What:** Per user decision, LLM-returned epistemic types must be strictly validated. Invalid types cause the entire extraction to be rejected.
**When to use:** In the extraction schema and materialization pipeline.

```python
# In ingestion/schema.py ExtractedFact:
from pydantic import field_validator

class ExtractedFact(BaseModel):
    epistemic_type: str = Field(
        default="asserted",
        description="Epistemic classification: observed, asserted, inferred, "
                    "hypothetical, conditional, or unverified"
    )

    @field_validator("epistemic_type")
    @classmethod
    def validate_epistemic_type(cls, v: str) -> str:
        allowed = {"observed", "asserted", "inferred", "hypothetical",
                   "conditional", "unverified"}
        if v.lower() not in allowed:
            raise ValueError(
                f"Invalid epistemic_type '{v}'. Must be one of: {allowed}. "
                "DEPRECATED is not assignable at creation."
            )
        return v.lower()
```

### Anti-Patterns to Avoid
- **Shifting column indices:** Never insert new columns in the middle of the CREATE TABLE definition. Always append. Use explicit SELECT lists if you need different ordering.
- **Silent type coercion:** Do not silently convert invalid epistemic types to a default. User decision mandates strict validation (reject entire extraction on invalid type).
- **Confidence recalculation during migration:** User decision explicitly states existing confidence values are preserved during backfill. Only epistemic_type is assigned.
- **getattr fallback after field exists:** Once epistemic_type is a native MemoryNode field, the getattr patterns in filtering.py and scoring.py become dead code that masks bugs. Remove them.

## Common Pitfalls

### Pitfall 1: DuckDB ALTER TABLE NOT NULL Limitation
**What goes wrong:** DuckDB 1.4.x does not support `ALTER TABLE ADD COLUMN ... NOT NULL`.
**Why it happens:** DuckDB's ALTER TABLE is limited compared to the CREATE TABLE DDL.
**How to avoid:** Use `DEFAULT 'asserted'` without NOT NULL on ALTER TABLE. The CREATE TABLE for new databases can have `NOT NULL DEFAULT 'asserted'`. This is the exact pattern used for the `scope` column migration in Phase 02.1.
**Warning signs:** If you see `Parser Error: NOT NULL` in ALTER TABLE, you've hit this.

### Pitfall 2: _row_to_node Column Index Drift
**What goes wrong:** Adding columns via ALTER TABLE appends them at the end. But `SELECT *` returns columns in table-definition order. If you also modify the CREATE TABLE to put new columns in a "logical" position, new databases and migrated databases will have different column orderings.
**Why it happens:** ALTER TABLE ADD COLUMN always appends; CREATE TABLE can put columns anywhere.
**How to avoid:** Add new columns in both ALTER TABLE and CREATE TABLE at the same position -- the END of the column list. Update `_row_to_node` to read from indices 16 and 17. If the column might not exist (pre-migration), use `len(row)` check or try/except for graceful degradation.
**Warning signs:** Existing tests pass but new field values are wrong (reading wrong column).

### Pitfall 3: Pydantic Model Frozen=False but ConfigDict Frozen=True Interaction
**What goes wrong:** MemoryObject has `ConfigDict(frozen=False)` and MemoryNode inherits it. The test suite uses `object.__setattr__` to set epistemic_type. After adding the field natively, Pydantic will handle it normally, but any code still using `object.__setattr__` will bypass validation.
**Why it happens:** The forward-compatible hack in tests must be removed when the field becomes native.
**How to avoid:** After adding the field, search for all `object.__setattr__` and `getattr(..., "epistemic_type"` patterns and replace them with direct field usage.
**Warning signs:** `grep -r "epistemic_type" --include="*.py"` should find NO getattr/setattr patterns after this phase.

### Pitfall 4: Extraction Prompt Token Budget
**What goes wrong:** Adding epistemic type classification to the extraction prompt increases prompt length, potentially pushing near token limits for long input messages.
**Why it happens:** The system prompt grows with each new extraction requirement.
**How to avoid:** Keep the epistemic type instructions concise. The existing prompt is ~500 tokens. Adding epistemic classification should add no more than ~100 tokens. Monitor extraction failures after the prompt change.
**Warning signs:** Increased extraction failure rate after prompt update.

### Pitfall 5: Strict Validation Rejects Valid Extractions
**What goes wrong:** The user decision mandates rejecting entire extractions when epistemic_type is invalid. This could cause data loss if the LLM frequently returns invalid types.
**Why it happens:** LLMs may return types not in the allowed set (e.g., "fact" instead of "asserted", "speculation" instead of "hypothetical").
**How to avoid:** Make the prompt extremely explicit about allowed values. Use instructor's retry mechanism (already configured with `max_retries=3`) -- instructor will re-prompt the LLM when Pydantic validation fails. The strict validation is on the Pydantic model, which instructor validates before returning.
**Warning signs:** Extraction success rate drops after adding epistemic_type to the schema.

### Backfill Heuristic (recommended)

```python
def infer_epistemic_type(node: MemoryNode) -> EpistemicType:
    """Best-guess epistemic type for pre-existing nodes without the field.

    Heuristics based on node_type and metadata:
    - ENTITY nodes -> ASSERTED (extracted from text)
    - FACT/DECISION/PREFERENCE nodes -> ASSERTED (LLM-extracted)
    - SUMMARY nodes -> INFERRED (system-generated)
    - NOTE nodes -> ASSERTED (user-provided via store())
    - EVENT nodes -> OBSERVED (direct user input)
    """
    if node.node_type == NodeType.SUMMARY:
        return EpistemicType.INFERRED
    if node.node_type == NodeType.EVENT:
        return EpistemicType.OBSERVED
    return EpistemicType.ASSERTED
```

## Open Questions

1. **CONDITIONAL epistemic type handling at ingestion**
   - What we know: CONDITIONAL is a valid creation-time type per user decision. RFC-0003 S5 requires condition, condition_scope, condition_state, evaluated_at fields.
   - What's unclear: EPIS-06 (full CONDITIONAL support) is deferred to Phase 5. Should Phase 3.1 add the type to the enum/field but not implement the condition tracking fields?
   - Recommendation: Include CONDITIONAL in the allowed creation-time types but do NOT add condition tracking fields in this phase. Nodes created as CONDITIONAL in this phase would lack condition evaluation -- acceptable since EPIS-06 is Phase 5. Document this as a known limitation.

2. **UNVERIFIED threshold in DEFAULT retrieval**
   - What we know: RFC-0003 S8 says UNVERIFIED is "Excluded unless above threshold" with default threshold `confidence > 0.30`. Current filtering.py only excludes HYPOTHETICAL and DEPRECATED.
   - What's unclear: Should Phase 3.1 add UNVERIFIED threshold filtering? EPIS-05 says "DEFAULT and EXPLICIT modes with per-epistemic-type inclusion rules" which includes the UNVERIFIED threshold.
   - Recommendation: Add UNVERIFIED threshold filtering to `filter_epistemic` as part of EPIS-05 completion. The threshold should be configurable (default 0.30). This is a small addition to the filtering function that completes the RFC-0003 S8 spec.

3. **Source type for MemoryEngine.store() path**
   - What we know: `store()` creates nodes directly without LLM extraction. The caller provides content and basic metadata but no source_type.
   - What's unclear: What source_type should `store()` default to?
   - Recommendation: Default to `USER_STATED` for `store()` since it's the direct API path. Allow caller override via optional `source_type` parameter. The `ingest()` path determines source_type from the extraction context (role-based: user messages -> USER_STATED, assistant messages -> SYSTEM_INFERRED).
