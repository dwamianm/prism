## Summary

This phase closes the three integration gaps (GAP-01, GAP-02, GAP-03) identified in the v1.0 audit and resolves the three CTXP traceability orphans. The work is purely connective — the underlying implementations for every feature already exist; the bugs are disconnected wiring in the call chains. No new abstractions, no new dependencies, no new algorithms.

The code changes are small: four call sites in `pipeline.py` and `engine.py` need one or two params added, and two `__init__` signatures need new optional params. The test work is larger: three E2E integration tests (one per gap) plus a milestone assertion. The traceability work is documentation-only: updating VERIFICATION.md and checking boxes in REQUIREMENTS.md.

GAP-02 is the single highest-leverage change: one line in `pipeline.py:_materialize()` that passes `temporal_intent=fact.temporal_intent` to `detect_and_supersede()`. Without it, the "assertion" branch in SupersedenceDetector is unreachable through automated ingestion — every ingested contradiction silently falls through to supersedence instead of creating a CONTRADICTS edge.

**Primary recommendation:** Fix the three gaps in dependency order (GAP-01, GAP-02, GAP-03), write one E2E integration test per gap using real backends in tmp_path fixtures, then update traceability docs. Commit in two plans: plan-01 (code fixes + tests) and plan-02 (traceability docs + milestone assertion).

### Established Pattern for Adding Optional Params

All existing optional params in this codebase follow the same pattern — confirmed across `filter_epistemic()`, `score_and_rank()`, `detect_and_supersede()`:

```python
# Pattern: None default + module-level constant fallback
def some_function(
    required_arg: str,
    optional_param: SomeType | None = None,
) -> Result:
    effective_param = (
        optional_param if optional_param is not None
        else MODULE_LEVEL_DEFAULT

    # use effective_param
```

This is precisely what `filter_epistemic()` already does for `unverified_threshold` and what `score_and_rank()` does for `epistemic_weights` — they accept `None` and fall back to module constants. The fix is to pass the values in rather than accepting None.

## Common Pitfalls

### Pitfall 1: Lambda Late Binding in write_queue.submit()
**What goes wrong:** `lambda: self._lexical_index.index(eid, c, uid, nt)` captures `scope` from the enclosing scope, but if the loop variable `scope` changes before the lambda executes, wrong value is used.
**Why it happens:** Python closures capture by reference, not by value.
**How to avoid:** Add `sc=scope.value` to the lambda default args: `lambda fid=fact_node_id, fc=fact_content, uid=event.user_id, nt=node_type.value, sc=fact_scope.value: (self._lexical_index.index(fid, fc, uid, nt, sc))`
**Warning signs:** Existing working examples in pipeline.py already use this pattern correctly (e.g., line 307 captures `txt=entity_text`, line 411 captures `fc=fact_content`).

### Pitfall 2: Scope Enum vs String
**What goes wrong:** `LexicalIndex.index()` expects `scope: str | None`. `Scope.PERSONAL` is an enum, not a string.
**Why it happens:** The enum is `(str, Enum)` so `Scope.PERSONAL == "personal"` is True, but tantivy's document builder expects an actual string.
**How to avoid:** Pass `scope.value` explicitly (e.g., `fact_scope.value`, `event.scope.value`). The existing working call in `test_retrieval_filter_forwarding.py:_store_node()` already does this correctly: `lexical_index.index(node_id, node.content, node.user_id, node.node_type.value, node.scope.value)`.

### Pitfall 3: RetrievalPipeline Backward Compatibility
**What goes wrong:** Adding required params to `RetrievalPipeline.__init__()` breaks existing callers in tests that construct it directly (e.g., `RetrievalPipeline(graph_store=..., vector_index=..., lexical_index=..., conn=...)`).
**Why it happens:** Forgetting to make new params optional with `= None`.
**How to avoid:** Per locked decision: `epistemic_weights: dict[str, float] | None = None` and `unverified_confidence_threshold: float | None = None`. Fall back to module-level defaults when None. Tests in test_retrieval_filter_forwarding.py construct `RetrievalPipeline` directly without new params — they must continue to work unchanged.

### Pitfall 4: Filter/Scoring Functions Already Accept None
**What goes wrong:** Misreading `filter_epistemic()` and `score_and_rank()` as needing signature changes.
**Why it happens:** The functions already have the right signatures. Confirmed:
  - `filter_epistemic(candidates, mode, unverified_threshold=None)` — if None, uses `UNVERIFIED_CONFIDENCE_THRESHOLD` constant
  - `score_and_rank(candidates, weights, epistemic_weights=None)` — if None, uses `EPISTEMIC_WEIGHTS` dict
  The fix is only in `pipeline.py:retrieve()` — pass stored values instead of accepting None default.
**How to avoid:** Only modify `pipeline.py` to pass `self._epistemic_weights` and `self._unverified_confidence_threshold` to these two calls.

### Pitfall 5: CTXP Requirements Already Implemented
**What goes wrong:** Treating CTXP-01/02/03 as needing code changes.
**Why it happens:** They appear in the phase requirements list and in REQUIREMENTS.md as `[ ]` unchecked.
**How to avoid:** These are traceability orphans only. The implementation exists in `src/prme/retrieval/packing.py` under RETR-06. The fix is documentation-only: update the Phase 3 VERIFICATION.md to add CTXP-01/02/03 evidence entries, and check their boxes in REQUIREMENTS.md.

### Pitfall 6: Summary Scope in pipeline.py
**What goes wrong:** Passing the wrong scope to the summary lexical write.
**Why it happens:** The summary write at lines 462-464 uses `event_id` and `result.summary` — the summary belongs to the event, not to any individual fact. The correct scope here is the ingestion-level `scope` param (not `fact_scope`).
**How to avoid:** Use `scope.value` (the method-level `scope: Scope` param) for the summary write, since summaries are event-level content.

@pytest.mark.asyncio
async def test_e2e_scope_isolation(backends):
    """Lexical scope filtering returns only matching-scope documents.

    GAP-03 regression: lexical_index.index() must receive scope so
    scope-filtered lexical queries exclude wrong-scope documents.

        "retrospective", "user1", scope=["project"]

    assert project_id in result_ids
    assert personal_id not in result_ids
