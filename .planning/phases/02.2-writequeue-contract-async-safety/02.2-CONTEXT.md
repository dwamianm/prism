# Phase 2.2: WriteQueue Contract & Async Safety - Context

**Gathered:** 2026-02-19
**Status:** Ready for planning

<domain>
## Phase Boundary

All graph writes during ingestion are serialized through WriteQueue. The embedding provider works safely in async contexts. Concurrent ingestion requests do not produce transaction conflicts or data loss. This phase closes audit gaps for INGE-05 (WriteQueue bypass) and integration issues #1 and #4.

</domain>

<decisions>
## Implementation Decisions

### Contract enforcement
- Structural prevention: after fixing the bypasses, direct GraphStore write access must be structurally prevented for ingestion components (not just documented convention)
- EntityMerger and SupersedenceDetector must not be able to accidentally call GraphStore write methods directly
- Claude's Discretion: whether reads stay direct or also go through a gateway — pick based on current patterns and complexity
- Claude's Discretion: whether GraphStore's internal _write_lock is removed or kept as safety net
- Claude's Discretion: injection approach (WriteQueue directly vs. restricted write interface) — pick what fits existing dependency patterns

### Embedding async strategy
- Change EmbeddingProvider protocol from synchronous to async — embed() becomes async
- OpenAIEmbeddingProvider: native async (already uses AsyncOpenAI internally)
- FastEmbedProvider: wrap sync ONNX calls in asyncio.to_thread()
- VectorIndex: make fully async, await embed() directly instead of current sync-with-thread pattern
- Add basic in-memory LRU cache for embedding results (avoid re-embedding identical text)

### Failure & rollback behavior
- Fail the entire event if any write fails — no partial state
- Roll back already-written graph nodes/edges from a failed event — clean state, no orphans
- Structured error types: custom exception hierarchy (WriteQueueError, EntityMergeError, etc.) with failed operation context
- Claude's Discretion: automatic retry policy — pick based on what failure modes are realistic for DuckDB/Kùzu writes

### Concurrency scope
- Claude's Discretion: concurrency level for v1.0 — pick based on API server design and realistic usage patterns
- Include concurrency integration tests that simulate parallel ingestion and verify no data loss
- Claude's Discretion: test scale (parallel event count) — pick what catches real bugs without being slow
- Concurrency tests use mock embeddings for CI, with optional real-embedding test for local validation with API key

### Claude's Discretion
- Read access pattern (direct vs. gateway) for ingestion components
- GraphStore internal _write_lock retention
- WriteQueue vs. restricted write interface for dependency injection
- Automatic retry policy for transient write failures
- Concurrency level and test scale for v1.0

</decisions>

<specifics>
## Specific Ideas

- Rollback must clean up graph artifacts from failed events — no orphan nodes/edges
- Embedding cache should be simple LRU, added while touching the embedding layer
- Concurrency tests should have both fast mock path (CI) and optional real-embedding path (local with API key)

</specifics>
