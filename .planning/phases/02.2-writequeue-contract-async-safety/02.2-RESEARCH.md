# Phase 2.2: WriteQueue Contract & Async Safety - Research

**Researched:** 2026-02-19
**Domain:** Write serialization, async embedding, concurrency safety for DuckDB-backed ingestion pipeline
**Confidence:** HIGH

## Summary

This phase closes two audit gaps: EntityMerger and SupersedenceDetector bypassing WriteQueue (INGE-05 partial), and OpenAIEmbeddingProvider using `asyncio.run()` which fails when called from an async context (integration issue #4). The codebase changes are well-scoped: (1) route entity/supersedence graph writes through WriteQueue, (2) structurally prevent future bypass via a restricted write interface, (3) convert the EmbeddingProvider protocol from synchronous to async, (4) add LRU caching for embeddings, and (5) implement rollback-on-failure for event writes with concurrency tests.

The technical domain is standard Python asyncio patterns. No new libraries are required -- everything uses stdlib asyncio, the existing OpenAI AsyncOpenAI client, and `asyncio.to_thread()` for blocking FastEmbed calls. The structural enforcement of write access is a Python Protocol / interface design problem, not a library problem.

**Primary recommendation:** Route all ingestion writes through WriteQueue by injecting a write-only interface (not the full GraphStore) into EntityMerger and SupersedenceDetector. Convert EmbeddingProvider to async. Use a simple `dict`-based LRU cache (or `functools.lru_cache` on a sync helper) for embeddings since the decision calls for in-memory LRU. Test concurrency with 5-10 parallel ingestion tasks using mock embeddings.

### Pattern 1: Restricted Write Interface via Protocol
**What:** Define a narrow Protocol that exposes only the write methods (create_node, create_edge, supersede) and inject this into EntityMerger/SupersedenceDetector instead of the full GraphStore.
**When to use:** When you need structural (type-system-enforced) prevention of direct write access.
**Why this over alternatives:** The user decision says "structurally prevent" -- a Protocol narrows the API surface at the type level. Mypy/pyright will flag any attempt to call read methods on the injected interface. At runtime, the implementation routes through WriteQueue.

```python
# Source: Python typing Protocol pattern (PEP 544)
from typing import Protocol

class GraphWriter(Protocol):
    """Write-only graph interface routed through WriteQueue."""
    async def create_node(self, node: MemoryNode) -> str: ...
    async def create_edge(self, edge: MemoryEdge) -> str: ...
    async def supersede(
        self, old_node_id: str, new_node_id: str, *, evidence_id: str | None = None
    ) -> None: ...
```

The implementation wraps WriteQueue.submit() calls:

```python
class WriteQueueGraphWriter:
    """GraphWriter that routes all writes through WriteQueue."""
    def __init__(self, graph_store: GraphStore, write_queue: WriteQueue) -> None:
        self._graph_store = graph_store
        self._write_queue = write_queue

    async def create_node(self, node: MemoryNode) -> str:
        return await self._write_queue.submit(
            lambda n=node: self._graph_store.create_node(n),
            label=f"graph.create_node:{node.id}",
        )

    async def create_edge(self, edge: MemoryEdge) -> str:
        return await self._write_queue.submit(
            lambda e=edge: self._graph_store.create_edge(e),
            label=f"graph.create_edge:{edge.id}",
        )

    async def supersede(self, old_node_id, new_node_id, *, evidence_id=None):
        await self._write_queue.submit(
            lambda: self._graph_store.supersede(old_node_id, new_node_id, evidence_id=evidence_id),
            label=f"graph.supersede:{old_node_id}->{new_node_id}",
        )
```

**Discretion recommendation (reads):** Keep reads direct on GraphStore. EntityMerger needs `query_nodes()` for lookup; SupersedenceDetector needs `get_edges()` and `get_node()`. Reads don't need serialization (DuckDB supports concurrent reads). Injecting both a `GraphReader` (for reads) and `GraphWriter` (for writes) cleanly separates concerns without over-engineering. However, just passing GraphStore for reads and GraphWriter for writes is sufficient -- a formal GraphReader protocol is unnecessary complexity for two read methods.

### Pattern 2: Async EmbeddingProvider Protocol
**What:** Change the `EmbeddingProvider` Protocol's `embed()` method from sync to async.
**When to use:** When the provider may do async I/O (OpenAI API) or blocking I/O (ONNX inference).

```python
@runtime_checkable
class EmbeddingProvider(Protocol):
    @property
    def model_name(self) -> str: ...
    @property
    def model_version(self) -> str: ...
    @property
    def dimension(self) -> int: ...

    async def embed(self, texts: list[str]) -> list[list[float]]: ...
```

**OpenAI implementation:** Remove `asyncio.run()`, call `_embed_async()` directly:
```python
class OpenAIEmbeddingProvider:
    async def embed(self, texts: list[str]) -> list[list[float]]:
        client = self._ensure_client()
        response = await client.embeddings.create(
            model=self._model_name, input=texts, dimensions=self._dimension,
        )
        return [item.embedding for item in response.data]
```

**FastEmbed implementation:** Wrap sync ONNX in `asyncio.to_thread()`:
```python
class FastEmbedProvider:
    async def embed(self, texts: list[str]) -> list[list[float]]:
        return await asyncio.to_thread(self._embed_sync, texts)

    def _embed_sync(self, texts: list[str]) -> list[list[float]]:
        self._ensure_model()
        assert self._model is not None
        return [embedding.tolist() for embedding in self._model.embed(texts)]
```

**VectorIndex changes:** Remove `asyncio.to_thread()` wrapping around embed calls. Call `await self._provider.embed([content])` directly:
```python
async def index(self, node_id: str, content: str, user_id: str) -> int:
```

### Pattern 3: Embedding LRU Cache
**What:** Simple in-memory LRU cache for embedding results, keyed by text content hash.
**When to use:** When the same text is embedded multiple times (entity names, repeated facts).

**Recommended approach:** Use a bounded `dict` with LRU eviction, or wrap with `functools.lru_cache` on a sync helper. Since `embed()` is now async, the cache layer should be in the async method, using a simple dict:

```python
import hashlib
from collections import OrderedDict

class CachedEmbeddingProvider:
    """Wrapper that adds LRU cache to any EmbeddingProvider."""
    def __init__(self, provider: EmbeddingProvider, maxsize: int = 512) -> None:
        self._provider = provider
        self._cache: OrderedDict[str, list[float]] = OrderedDict()
        self._maxsize = maxsize

    async def embed(self, texts: list[str]) -> list[list[float]]:
        results: list[list[float]] = []
        uncached_texts: list[str] = []
        uncached_indices: list[int] = []

        for i, text in enumerate(texts):
            key = hashlib.sha256(text.encode()).hexdigest()
            if key in self._cache:
                self._cache.move_to_end(key)
                results.append(self._cache[key])
            else:
                results.append([])  # placeholder
                uncached_texts.append(text)
                uncached_indices.append(i)

        if uncached_texts:
            embeddings = await self._provider.embed(uncached_texts)
            for idx, emb in zip(uncached_indices, embeddings):
                key = hashlib.sha256(texts[idx].encode()).hexdigest()
                self._cache[key] = emb
                if len(self._cache) > self._maxsize:
                    self._cache.popitem(last=False)
                results[idx] = emb

        return results
```

**Alternative:** For FastEmbed (sync path only), could use `functools.lru_cache` on `_embed_sync` for single-text calls. But the wrapper approach is cleaner because it works for both providers.

### Pattern 4: Rollback on Failure
**What:** Track graph writes within an event's materialization and undo them if any step fails.
**When to use:** When the user decision requires "no partial state" on failure.

```python
class WriteTracker:
    """Tracks graph write operations for rollback."""
    def __init__(self) -> None:
        self._created_node_ids: list[str] = []
        self._created_edge_ids: list[str] = []

    def record_node(self, node_id: str) -> None:
        self._created_node_ids.append(node_id)

    def record_edge(self, edge_id: str) -> None:
        self._created_edge_ids.append(edge_id)

    async def rollback(self, graph_store: GraphStore) -> None:
        """Delete all tracked writes in reverse order."""
        # Edges first, then nodes (referential integrity)
        for edge_id in reversed(self._created_edge_ids):
            try:
                await graph_store.delete_edge(edge_id)
            except Exception:
                logger.error("rollback.edge_failed", edge_id=edge_id)
        for node_id in reversed(self._created_node_ids):
            try:
                await graph_store.delete_node(node_id)
            except Exception:
                logger.error("rollback.node_failed", node_id=node_id)
```

**Important:** DuckDB does not have built-in graph-level transactions for this. The rollback must be application-level: track what was created, delete on failure. This requires adding `delete_node()` and `delete_edge()` methods to GraphStore. These are cleanup-only operations, not part of normal flow.

### Anti-Patterns to Avoid
- **Passing full GraphStore to ingestion components:** This is the exact problem being fixed. EntityMerger should never have direct access to `graph_store.create_node()`.
- **Using `nest_asyncio` to fix the OpenAI embed issue:** Global event loop patching is fragile and masks the real problem. The correct fix is making embed() async.
- **Caching embeddings by text without model versioning:** If the model changes, cached embeddings are invalid. The cache should be invalidated when the provider changes (simplest: cache lives on the provider instance, so new provider = new cache).
- **Testing concurrency only with sequential assertions:** Concurrency bugs require actual parallel execution. Tests must use `asyncio.gather()` with multiple ingestion calls.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async LRU cache | Complex concurrent-safe async cache with dedup | Simple `OrderedDict`-based cache or `functools.lru_cache` on sync helper | Embedding calls are not highly concurrent (serialized through WriteQueue); simple cache sufficient |
| Nested event loop fix | Custom event loop management or `nest_asyncio` | Native async `embed()` method | Clean solution per user decision; no runtime hacks |
| Graph transaction rollback | Custom transaction manager | Simple write tracker with compensating deletes | DuckDB graph operations are individual INSERTs; compensating deletes are straightforward |
| Write serialization | Custom locking per component | Existing `WriteQueue` with `submit()` | WriteQueue already provides sequential execution; just route all writes through it |

**Key insight:** The existing WriteQueue already solves serialization. The only problem is that some callers bypass it. The fix is plumbing (routing), not new infrastructure.

## Common Pitfalls

### Pitfall 1: asyncio.run() Inside Running Event Loop
**What goes wrong:** `OpenAIEmbeddingProvider.embed()` currently calls `asyncio.run()` which raises `RuntimeError: asyncio.run() cannot be called from a running event loop` when called from an async context. It only works today because VectorIndex wraps it in `asyncio.to_thread()`, which creates a new thread without an event loop.
**Why it happens:** The original design made `embed()` sync per Protocol, then needed to call async OpenAI API, so used `asyncio.run()` as escape hatch.
**How to avoid:** Make `embed()` async in the Protocol. OpenAI uses native `await`. FastEmbed uses `asyncio.to_thread()`.
**Warning signs:** `RuntimeError` when any code path calls `embed()` from async context without `to_thread()`.

### Pitfall 2: Read-Write Ordering with WriteQueue
**What goes wrong:** EntityMerger reads existing entities (via `query_nodes()`), then creates a new one if not found. If two concurrent ingestion calls process the same entity name, both might read "not found" and both create duplicates.
**Why it happens:** The read (query_nodes) happens outside the WriteQueue. The write (create_node) goes through WriteQueue. There's a TOCTOU (time-of-check-to-time-of-use) gap.
**How to avoid:** For v1.0, this is acceptable because: (a) concurrent ingestion of identical entities from different events is rare, (b) the entity merger is conservative (duplicates are better than incorrect merges per project decision), (c) DuckDB single-writer serialization through WriteQueue prevents data corruption. For v2.0, consider making the entire find_or_create_entity atomic via a single WriteQueue job.
**Warning signs:** Duplicate entity nodes with identical name+type appearing in the graph.

### Pitfall 3: Rollback Leaving Vector/Lexical Artifacts
**What goes wrong:** Rollback cleans up graph nodes and edges, but vectors and lexical index entries for those nodes remain. Subsequent searches return IDs pointing to deleted graph nodes.
**How to avoid:** Rollback must also remove vector and lexical entries. Track all three stores' artifacts in the WriteTracker: node IDs, edge IDs, and indexed content IDs. Lexical and vector cleanup can be best-effort (logged warning on failure) since they are derived and rebuildable.
**Warning signs:** Search returns node_ids that produce `None` when looked up in graph store.

### Pitfall 4: Embedding Cache Invalidation on Model Change
**What goes wrong:** Cache returns embeddings from old model after provider configuration changes.
**Why it happens:** Cache is keyed by text content only, not by model name/version.
**How to avoid:** The cache lives on the provider wrapper instance. When the engine is reconfigured with a new model, a new provider instance is created, which gets a new empty cache. No cross-model contamination.
**Warning signs:** Cosine similarity scores degrade after model configuration change without restart.

### Pitfall 5: DuckDB Connection Used Across Threads Without Care
**What goes wrong:** DuckDB connections are not thread-safe in Python. If `asyncio.to_thread()` is used for graph writes while the main thread also uses the same connection, corruption can occur.
**Why it happens:** `DuckPGQGraphStore._write_lock` protects async callers but doesn't protect against thread-level races when `to_thread()` is used.
**How to avoid:** The WriteQueue serializes all writes through a single consumer task. Since the consumer runs sequentially, only one write happens at a time, eliminating thread races. The existing `_write_lock` in DuckPGQGraphStore is redundant when all writes go through WriteQueue, but keeping it as a safety net (per discretion area) adds defense-in-depth.
**Warning signs:** Sporadic `duckdb.CatalogException` or data corruption under concurrent load.

## Discretion Recommendations

### Read access pattern: Keep direct reads
**Recommendation:** Ingestion components keep direct `GraphStore` reference for reads. No read gateway needed.
**Rationale:** DuckDB supports concurrent reads. Reads don't need serialization. Adding a read gateway would add complexity without benefit. EntityMerger and SupersedenceDetector both need `query_nodes()`, `get_edges()`, and `get_node()` for their detection logic -- these are read-only operations.

### GraphStore internal _write_lock: Keep as safety net
**Recommendation:** Keep `DuckPGQGraphStore._write_lock` but add a comment marking it as "defense-in-depth, primary serialization is via WriteQueue."
**Rationale:** If a future code path accidentally calls GraphStore directly (bypassing WriteQueue), the lock prevents corruption. Removing it saves zero performance (asyncio.Lock is essentially free when uncontended) and removes a safety layer.

### Injection approach: WriteQueueGraphWriter (restricted write interface)
**Recommendation:** Create a `GraphWriter` Protocol and `WriteQueueGraphWriter` implementation. Inject `GraphWriter` into EntityMerger and SupersedenceDetector instead of raw GraphStore.
**Rationale:** User decision requires "structural prevention." A Protocol enforces the API surface at the type level. Mypy/pyright will catch any attempt to call read methods on a `GraphWriter`. The `WriteQueueGraphWriter` implementation routes through WriteQueue, ensuring serialization.

### Retry policy: No automatic retry for graph writes
**Recommendation:** Do not add automatic retry for graph write failures. Fail the event, rollback, and let the existing extraction retry mechanism (with exponential backoff) handle retries at the event level.
**Rationale:** Graph write failures in DuckDB are typically deterministic (constraint violations, schema errors). Retrying the same write will produce the same error. The existing `_schedule_retry()` mechanism in IngestionPipeline already handles transient failures (LLM API errors, timeouts) at the event level. Retrying at the individual write level adds complexity without clear benefit.

### Concurrency level for v1.0: 5-10 parallel tasks in tests
**Recommendation:** Test with 5 parallel ingestion calls (fast CI), optional 10-task test for thorough validation. The WriteQueue maxsize=1000 handles far more; the test scale is about catching real bugs, not load testing.
**Rationale:** 5 parallel tasks is enough to trigger race conditions (TOCTOU in entity merge, concurrent WriteQueue consumers). More than 10 adds test runtime without catching additional bug categories. Real load testing is a Phase 4 concern (HTTP API).

## Open Questions

1. **GraphStore.delete_node() and delete_edge() methods**
   - What we know: Rollback needs to delete graph artifacts from failed events. The current GraphStore Protocol has no delete methods.
   - What's unclear: Whether delete should be a full Protocol method or a separate cleanup interface.
   - Recommendation: Add `delete_node(node_id: str)` and `delete_edge(edge_id: str)` to GraphStore Protocol and DuckPGQGraphStore. These are needed for rollback and will also be needed for archival (Phase 5). Route through WriteQueue like other writes.

2. **Vector/Lexical cleanup on rollback**
   - What we know: LexicalIndex.delete_by_node_id() is a stub (Phase 1 tech debt). VectorIndex has no delete method.
   - What's unclear: How to cleanly remove vectors from USearch (USearch supports removal but it's not exposed in VectorIndex).
   - Recommendation: For this phase, implement rollback for graph artifacts only. Log a warning for orphaned vector/lexical entries. Add a TODO for vector/lexical cleanup when those delete methods are implemented. This is acceptable because vector/lexical entries pointing to deleted graph nodes are harmless (search returns an ID, get_node returns None, caller handles gracefully).

3. **Embedding cache scope**
   - What we know: User wants "basic in-memory LRU cache for embedding results."
   - What's unclear: Cache scope -- per-provider-instance? Per-engine-lifetime? Shared across users?
   - Recommendation: Per-provider-instance cache (CachedEmbeddingProvider wraps the real provider). This means cache lives for the engine lifetime and is not shared across separate MemoryEngine instances. Cache is keyed by text content, not user_id (embeddings are user-agnostic). Maxsize of 512 entries is reasonable (covers typical entity names + fact texts for a session).
