# Phase 2.1: Scope Isolation Fix - Research

**Researched:** 2026-02-19
**Domain:** DuckDB schema migration, Python data pipeline threading, LLM extraction schema modification
**Confidence:** HIGH

## Summary

Phase 2.1 is a focused gap-closure phase that addresses two audit integration issues: (1) `Event.scope` is defined on the `MemoryObject` base model but silently dropped when persisting to DuckDB because the `events` table has no `scope` column, and (2) `IngestionPipeline.ingest()` has no `scope` parameter so all extracted `MemoryNode` objects are hardcoded to `Scope.PERSONAL`.

The fix is surgical: add a `scope` column to the `events` table in DuckDB (with migration for existing data), thread a `scope` parameter through `IngestionPipeline.ingest()` into the materialization logic, update the extraction schema so the LLM classifies scope per extracted object, and add scope filtering to `EventStore` queries. No new libraries are needed. No architectural changes are required. The existing `Scope` enum (`PERSONAL`, `PROJECT`, `ORG`) and `MemoryNode.scope` field are already correct -- the gap is purely in persistence and pipeline plumbing.

**Primary recommendation:** Add `scope VARCHAR NOT NULL DEFAULT 'personal'` to the events table, use DuckDB `ALTER TABLE ADD COLUMN` for auto-migration at startup, thread `scope: Scope = Scope.PERSONAL` through all ingestion and query paths, and update the LLM extraction prompt/schema to classify scope per object.

### Pattern 1: Startup Auto-Migration via ALTER TABLE ADD COLUMN

**What:** Detect missing `scope` column on the `events` table at startup and add it with a default value, backfilling all existing rows as `'personal'`.

**When to use:** Schema evolution in a single-user embedded database where formal migration tooling is overkill.

**How DuckDB handles this:** DuckDB's `ALTER TABLE ... ADD COLUMN ... DEFAULT` sets the default for existing rows. This is an atomic operation. For idempotency, check `INFORMATION_SCHEMA.COLUMNS` first.

**Confidence:** HIGH -- DuckDB ALTER TABLE ADD COLUMN with DEFAULT is well-documented and backfills existing rows atomically.

```python
# In schema.py, add migration function called from initialize_database()
def _migrate_events_scope(conn: duckdb.DuckDBPyConnection) -> None:
    """Add scope column to events table if missing (auto-migration)."""
    result = conn.execute("""
        SELECT column_name FROM information_schema.columns
        WHERE table_name = 'events' AND column_name = 'scope'
    """).fetchone()
    if result is None:
        conn.execute("""
            ALTER TABLE events
            ADD COLUMN scope VARCHAR NOT NULL DEFAULT 'personal'
        """)
        conn.execute(
            "CREATE INDEX IF NOT EXISTS idx_events_scope ON events (scope)"
        )
        logger.info("Migrated events table: added scope column (backfilled as 'personal')")
```

### Pattern 2: Scope Threading Through Pipeline (Parameter Propagation)

**What:** Add `scope: Scope = Scope.PERSONAL` as a keyword argument to `IngestionPipeline.ingest()`, propagate it into `_materialize()`, and use it as the default scope for all created `MemoryNode` objects. The LLM extraction can override this per-object.

**When to use:** When a parameter needs to flow from an entry point through multiple layers of an ingestion pipeline.

**Key design decision -- scope as default fallback:** The ingestion-level scope acts as a **default fallback** (not a ceiling). The LLM extraction prompt classifies scope per object. If the LLM doesn't assign a scope (field is None or missing), the ingestion-level scope is used. This is safer than a ceiling because:
1. It preserves backward compatibility (callers who don't pass scope get PERSONAL as before).
2. It allows the LLM to upgrade scope when context warrants it (e.g., a message in a personal context mentions an org-wide decision).
3. RFC-0004 treats scope as a classification property, not a security boundary at this stage.

```python
# IngestionPipeline.ingest() signature change
async def ingest(
    self,
    content: str,
    *,
    user_id: str,
    role: str = "user",
    session_id: str | None = None,
    scope: Scope = Scope.PERSONAL,  # NEW
    metadata: dict | None = None,
    wait_for_extraction: bool = False,
) -> str:
```

**What:** Add an optional `scope` field to `ExtractedEntity` and `ExtractedFact` in the extraction schema, and update the extraction system prompt to instruct the LLM on scope classification.

### Pattern 5: Graph Node Scope Verification Migration

**What:** At startup, check existing graph nodes for scope consistency and backfill any missing scope values.

**When to use:** Per user decision -- run a verification/update migration on existing graph nodes.

**Note:** The `nodes` table already has a `scope VARCHAR NOT NULL DEFAULT 'personal'` column. This migration is purely a verification step to confirm no NULLs slipped in. In practice, this should be a no-op since the column already has a NOT NULL constraint. A simple `SELECT COUNT(*) FROM nodes WHERE scope IS NULL` check suffices, with `UPDATE nodes SET scope = 'personal' WHERE scope IS NULL` as the safety net (which DuckDB will skip because NOT NULL is already enforced).

### Anti-Patterns to Avoid

- **Scope as a security boundary in this phase:** RFC-0004 defines full namespace isolation with access control, but this phase is about data classification, not access enforcement. Don't add permission checks -- that's future work.
- **Breaking backward compatibility:** Every existing call site that doesn't pass `scope` must continue to work identically. Always default to `Scope.PERSONAL`.
- **Storing scope as a separate table/entity:** Scope is a classification label on each object, not a first-class entity. A simple VARCHAR column is sufficient.
- **Making scope mutable on events:** Events are immutable (frozen=True). Once an event is stored with a scope, it cannot change. This aligns with the append-only design constraint.
- **Overcomplicating the LLM prompt:** The scope classification should be a simple additional instruction in the existing extraction prompt, not a separate extraction step.

## Common Pitfalls

### Pitfall 1: DuckDB ALTER TABLE ADD COLUMN on Existing Non-Empty Table

**What goes wrong:** Forgetting that ALTER TABLE ADD COLUMN with a DEFAULT value in DuckDB atomically backfills all existing rows. Developers sometimes add the column as nullable first, then try to UPDATE, then add NOT NULL -- this is unnecessary complexity.

**Why it happens:** Habit from databases where ADD COLUMN doesn't backfill.

**How to avoid:** Use `ALTER TABLE events ADD COLUMN scope VARCHAR NOT NULL DEFAULT 'personal'` directly. DuckDB handles the backfill atomically. Verify with `SELECT scope FROM events LIMIT 5` after migration.

**Warning signs:** Writing a two-step migration (ADD COLUMN nullable + UPDATE + ALTER COLUMN NOT NULL) when a single statement works.

### Pitfall 2: Event._row_to_event() Column Index Shift

**What goes wrong:** Adding a column to the events table changes the column indices used in `EventStore._row_to_event()`. If `scope` is added as a new column, it shifts the column order for `SELECT *` queries.

**Why it happens:** The current `_row_to_event()` uses positional tuple indexing (`row[0]`, `row[1]`, etc.) rather than named column access.

**How to avoid:** Either: (a) add scope at a specific position in the CREATE TABLE and update all indices, or (b) add scope at the end (DuckDB ALTER TABLE adds columns at the end) and update `_row_to_event()` to read scope from the new last position. Option (b) is simpler and aligns with how ALTER TABLE works. For new databases (fresh CREATE TABLE), put scope in the logical position (after session_id).

**Warning signs:** `Event` objects coming back with wrong field values after the migration. Always verify the column order matches the tuple indexing.

### Pitfall 3: Hardcoded Scope.PERSONAL in Multiple Locations

**What goes wrong:** Missing one of the several places where `Scope.PERSONAL` is hardcoded, leading to inconsistent scope behavior.

**Why it happens:** The hardcoded scope is scattered across multiple files: `pipeline.py` (fact node creation at line 302), `entity_merge.py` (new entity creation at line 99).

**How to avoid:** Grep for `Scope.PERSONAL` and `scope=Scope.PERSONAL` across the codebase. Every occurrence must either be updated to use the threaded scope parameter or explicitly documented as intentionally hardcoded.

**Warning signs:** Some nodes getting the correct scope from the pipeline while others always get PERSONAL.

### Pitfall 4: Extraction Schema Change Breaking Existing LLM Behavior

**What goes wrong:** Adding a `scope` field to the extraction schema causes the LLM to hallucinate scope values or changes the quality of other extracted fields.

**Why it happens:** Adding fields to a structured extraction schema changes the LLM's output distribution. instructor handles optional fields gracefully, but the prompt must clearly define what scope values mean.

**How to avoid:** Make the `scope` field optional (default=None). Update the extraction prompt to include clear examples. Use the ingestion-level scope as fallback when LLM returns None. Test that extraction quality for entities/facts/relationships is not degraded by the new field.

**Warning signs:** LLM consistently returning None for scope (prompt too vague) or always returning PERSONAL (prompt not descriptive enough about when to use PROJECT/ORG).

### Pitfall 5: EventStore Query Methods Missing Scope Parameter

**What goes wrong:** `EventStore.get_by_user()` and `EventStore.get_by_hash()` don't offer scope filtering, so downstream consumers can't scope event queries.

**Why it happens:** Forgetting that scope is a query dimension, not just a storage field.

**How to avoid:** Add optional `scopes: list[Scope] | None = None` parameter to query methods. When None, return all scopes (backward compatible). When provided, filter with `scope IN (...)`.

**Warning signs:** Callers retrieving all events for a user regardless of scope context.

## Discretion Recommendations

Based on RFC-0004 analysis and codebase consistency:

### Scope as Classification Label (not isolation boundary)

**Recommendation:** Treat scope as a **classification label** in this phase, not a security isolation boundary.

**Rationale:** RFC-0004 defines full namespace isolation with access policies, permission checks, and vector index partitioning. That is Phase 3+ work. This phase is about persisting the scope classification and threading it through the pipeline so that future phases can build isolation on top. Treating scope as an isolation boundary now would require implementing access control, which is out of scope.

### Entities Carry Scope

**Recommendation:** Entities carry scope, same as facts and relationships.

**Rationale:** The `MemoryNode` base model already has a `scope` field. Entities are `MemoryNode` instances. Removing scope from entities would require a model split or special-casing. Additionally, the same entity name can exist in different scopes (e.g., "Project Alpha" might be personal context in one scope and project context in another). RFC-0004 Section 2 defines that namespaces contain "memory objects, events, and entities" -- entities are scoped.

### Flat Independent Scopes (No Hierarchy)

**Recommendation:** Treat PERSONAL, PROJECT, and ORG as **flat independent buckets** with no inheritance.

**Rationale:** RFC-0004 Section 4 states "Permissions do NOT automatically inherit from parent to child." While RFC-0004 supports optional parent_id for namespace hierarchy, implementing hierarchy adds complexity (traversal, permission propagation) that is not needed for data classification. Multi-scope queries (already a locked decision) provide the needed flexibility without hierarchy. Hierarchy can be added later without breaking flat scopes.

### Scope Immutability on Events and Nodes

**Recommendation:** Scope is **immutable once assigned** to events. For nodes, scope can be updated via supersedence (create a new node with the corrected scope, supersede the old one).

**Rationale:** Events are frozen (immutable by design constraint). Node scope changes should create a new version rather than mutating in place, consistent with the append-only design and supersedence pattern already established.

### Ingestion Scope as Default Fallback

**Recommendation:** The ingestion-level scope acts as a **default fallback** when the LLM doesn't classify scope.

**Rationale:** Using it as a ceiling would prevent the LLM from correctly classifying org-wide facts mentioned in a personal conversation. A fallback is safer: if the LLM is unsure (returns None), use the caller's scope. If the LLM classifies explicitly, trust it. This preserves LLM extraction quality while providing a sensible default.

### Fallback Scope: PERSONAL

**Recommendation:** When the LLM doesn't classify scope (returns None/null), default to **PERSONAL**.

**Rationale:** PERSONAL is the most restrictive scope. Defaulting to a wider scope (PROJECT, ORG) risks leaking personal information into shared contexts. PERSONAL is also the current hardcoded behavior, so this maintains backward compatibility.

### Defer Vector/Lexical Scope Metadata to Phase 3

**Recommendation:** Do **not** add scope to vector_metadata or tantivy documents in this phase.

**Rationale:** Vector and lexical indexes currently filter by user_id only. Adding scope filtering requires changes to USearch post-filter logic, tantivy query construction, and potentially separate indexes per scope (per RFC-0004 Section 6 retrieval isolation requirements). This is retrieval-layer work that belongs in Phase 3. The scope data will be on the graph nodes and events -- Phase 3 can join against it.

### Basic Scope Filtering on EventStore: This Phase

**Recommendation:** Add scope filtering to `EventStore.get_by_user()` and `get_by_hash()` in this phase.

**Rationale:** The success criteria require "events can be queried by scope." EventStore is the simplest query surface and the scope column is right there in DuckDB. Not adding filtering would leave the new column unused for queries. However, defer scope filtering on `MemoryEngine.search()` to Phase 3 since that involves vector/lexical backends.

### Scope Does Not Influence Ranking (This Phase)

**Recommendation:** Scope does **not** influence ranking in this phase.

**Rationale:** Ranking is a Phase 3 retrieval concern. Multi-scope queries in this phase return results filtered by scope but ranked by existing criteria (timestamp for events, confidence/created_at for nodes). Scope-aware ranking (e.g., prefer personal over project) can be added as a scoring weight in Phase 3's configurable re-ranker.

## Open Questions

1. **Extraction quality impact of scope field**
   - What we know: Adding optional fields to instructor schemas generally works well; the LLM may return None frequently for ambiguous cases.
   - What's unclear: Whether GPT-4o-mini reliably classifies scope, or whether it defaults to None/personal most of the time.
   - Recommendation: Accept this as expected behavior. The fallback to ingestion-level scope handles it. Can be improved with few-shot examples in the prompt if needed.

2. **SELECT * vs explicit column lists after migration**
   - What we know: DuckDB ALTER TABLE adds columns at the end. The current code uses `SELECT *` with positional indexing.
   - What's unclear: Whether DuckDB guarantees column order for tables that have been altered versus freshly created.
   - Recommendation: Switch EventStore queries to use explicit column lists (`SELECT id, timestamp, role, content, ...`) instead of `SELECT *`. This eliminates positional dependency entirely and is safer for future migrations.
