**Scope parameter threaded through full ingestion pipeline with LLM-driven per-object scope classification and ingestion-level fallback**

## Accomplishments
- Added optional scope fields to ExtractedEntity and ExtractedFact extraction schemas
- Added Scope Classification instructions to EXTRACTION_SYSTEM_PROMPT guiding LLM to classify each object as personal/project/org
- Threaded scope parameter through MemoryEngine.ingest/ingest_batch -> IngestionPipeline.ingest/ingest_batch -> _extract_and_materialize -> _materialize -> EntityMerger.find_or_create_entity
- Implemented per-object scope resolution: LLM-extracted scope overrides ingestion-level default, with null fallback
- Eliminated all hardcoded Scope.PERSONAL from node creation code (only remains in default parameter values)
- Maintained full backward compatibility: existing callers without scope parameter get PERSONAL behavior

## Decisions Made
- LLM-extracted scope (per entity/fact) overrides ingestion-level scope when present; null falls back to ingestion-level default
- Scope param defaults to Scope.PERSONAL at all entry points for backward compatibility
- Entity merge lookup does not filter by scope (conservative merge: same entity in different scopes still merges within same user_id, per research recommendations)
- Scope is passed to Event constructor at ingestion time so the event itself carries the correct scope
