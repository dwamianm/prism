# Phase 3.2: Retrieval Filter Forwarding - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Forward scope and temporal filters to all retrieval backends (vector, lexical, graph) before candidate merging. Candidates from wrong scopes are excluded; expired nodes are excluded from candidates. This is filter plumbing — the filter logic itself (what scopes exist, how temporal validity works) is already defined.

</domain>

<decisions>
## Implementation Decisions

### Default filter behavior
- When retrieve() is called without scope, return results from all scopes (no filtering). Scope filtering is opt-in.
- When scope IS provided, accept a list of scopes (e.g., scope=[PROJECT, PERSONAL]) — not just a single value.
- Retrieval response should include metadata about which filters were active (scope list, temporal window) for debugging and explainability.

### Cross-scope visibility
- Scope filtering uses hard walls for primary results — scope=[PROJECT] returns only PROJECT candidates.
- Cross-scope hints: highly relevant results from other scopes appear separately, flagged as cross-scope. Only triggered if the result would have ranked in the top-N results ignoring scope (high bar, rare hints).
- Cross-scope hints are on by default. Caller can disable with include_cross_scope=False.
- Cross-scope hints respect the same temporal filters as primary results.

### Temporal boundary semantics
- Cross-scope hints respect temporal filters — consistent filtering behavior.

### Claude's Discretion
- Default temporal behavior when no temporal filter is provided (all-time vs currently-valid)
- Temporal boundary handling (hard cutoff vs grace window for recently-expired nodes)
- Whether temporal filtering applies uniformly to all node types or exempts long-lived types (Entities, Preferences)
- Whether temporal intent from QueryAnalysis constrains candidate generation or re-ranks after generation
- Cross-scope hint token budget allocation (separate vs shared with primary results)

</decisions>

<specifics>
## Specific Ideas

- Multi-scope list parameter enables flexible queries like "show me PROJECT and PERSONAL but not ORGANISATION"
- Cross-scope hints modeled after how search engines show "results from other categories" — rare, flagged, high-relevance-only
- Filter metadata in response aligns with existing explainable score trace pattern from Phase 3

</specifics>
