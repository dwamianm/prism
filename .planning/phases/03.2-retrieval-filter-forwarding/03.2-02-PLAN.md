**Cross-scope hint generation (src/prme/retrieval/pipeline.py):**

After the primary candidate generation (Stage 2-3) and before scoring (Stage 5), add cross-scope hint logic:

```python
# After primary candidates are generated:
cross_scope_hints = []
if scope is not None and include_cross_scope and candidates:
    # Secondary pass: vector + lexical only (cheapest backends), no scope filter.
    # Per research Pattern 3: only run cheapest backends with reduced k.
    hint_config = effective_packing_config.model_copy(
        update={"vector_k": effective_packing_config.cross_scope_top_n * 2,
                "lexical_k": effective_packing_config.cross_scope_top_n * 2}
    )
    # Run secondary generation without scope filter but WITH temporal filter
    hint_candidates, _ = await generate_candidates(
        analysis,
        graph_store=self._graph_store,
        vector_index=self._vector_index,
        lexical_index=self._lexical_index,
        user_id=user_id,
        scope=None,  # No scope filter for hints
        time_from=effective_time_from,
        time_to=effective_time_to,
        config=hint_config,
    )
    # Filter to only results NOT in primary scopes
    primary_scope_values = {s.value for s in scope}
    hint_candidates = [
        c for c in hint_candidates
        if c.node.scope.value not in primary_scope_values
    ]
    # Score the hints using the same weights
    if hint_candidates:
        scored_hints, _ = score_and_rank(hint_candidates, effective_weights)
        # Only include if they would rank in top-N of ALL unscoped results
        cross_scope_hints = scored_hints[:effective_packing_config.cross_scope_top_n]
```

Per research Pitfall 4, this only runs vector+lexical (not a full 4-backend pass). Use reduced k for performance. The cross_scope_top_n and cross_scope_token_budget config values from PackingConfig (added in Plan 01) control the limits.
