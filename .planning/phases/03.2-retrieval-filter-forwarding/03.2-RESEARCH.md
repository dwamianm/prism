# Phase 3.2: Retrieval Filter Forwarding - Research

**Researched:** 2026-02-21
**Domain:** Retrieval pipeline filter propagation (scope isolation + temporal filtering)
**Confidence:** HIGH

## Summary

This phase is internal plumbing work -- no new libraries, no new data models. The code changes are surgical and contained entirely within the existing retrieval and storage modules. The current codebase accepts `scope` and temporal parameters at the `MemoryEngine.retrieve()` and `RetrievalPipeline.retrieve()` entry points, and `QueryAnalysis` already carries `time_from`/`time_to` from Stage 1 analysis. However, `generate_candidates()` currently does NOT forward these filters to any of the four backend generators. The `scope` parameter is accepted in `generate_candidates()` but never passed down to the individual `_generate_*` functions. Temporal filters from `QueryAnalysis` are never used during candidate generation at all.

The work breaks into three clear parts: (1) forward scope filters to all four backend generators, (2) forward temporal filters to all four backend generators, (3) add cross-scope hint support and filter metadata to the response. The user has made clear decisions about multi-scope list support, cross-scope hints, and filter metadata that constrain the implementation.

**Primary recommendation:** Thread `scope: list[Scope] | None` and temporal window (`time_from`/`time_to`) through `generate_candidates()` to all four backend generators, modify each backend's search method to accept and enforce these filters, add cross-scope hint generation as a secondary retrieval pass, and attach filter metadata to `RetrievalResponse`.

### Pattern 1: Filter Forwarding Through Candidate Generation

**What:** Thread filter parameters from the pipeline orchestrator through generate_candidates() down to each individual backend generator function.

**When to use:** Any time a filter constraint needs to be enforced before candidate merging rather than after.

**Current state (the gap):**

### Pattern 3: Cross-Scope Hints via Secondary Retrieval Pass

**What:** After primary scoped retrieval, run a secondary pass without scope filter, collect top-N results that are NOT in the primary scope, and include them separately as cross-scope hints.

**When to use:** When the user requests scope-filtered retrieval but wants to see highly relevant results from other scopes as soft hints.

```python
# Pseudo-code for cross-scope hint generation:
primary_candidates = generate_candidates(..., scope=requested_scopes)
if include_cross_scope:
    all_candidates = generate_candidates(..., scope=None)  # No scope filter
    hint_candidates = [
        c for c in all_candidates
        if c.node.scope not in requested_scopes
        and c would rank in top N
    ]
    # Attach as separate list, NOT merged with primary
```

### Pattern 4: Temporal Filtering at Candidate Generation

**What:** Forward `time_from`/`time_to` from QueryAnalysis to backend generators to exclude expired nodes before merging.

**Graph backend:** Already supports `valid_at` parameter on `query_nodes()` and `get_neighborhood()`. Extend to support a range (`valid_from`/`valid_to` window) rather than just a point in time.

**Vector backend:** Join with nodes table to check `valid_from <= ? AND (valid_to IS NULL OR valid_to > ?)`.

**Lexical backend:** Post-filter after tantivy search (tantivy has no temporal field), or add temporal fields to tantivy schema. Post-filter is acceptable here since the filtering happens before candidate merging (the RFC requires pre-merge filtering, not necessarily at-index-level for lexical).

### Anti-Patterns to Avoid

- **Post-merge scope filtering:** Filtering by scope AFTER merging candidates from all backends violates NSPC-05 and RFC-0004 S6. The scope filter MUST be applied at each backend individually before candidates enter the merge stage.
- **Silently degrading to no-filter:** If a backend cannot enforce scope filtering, it must return an empty result set for that backend, not silently return unfiltered results. This is the "fail safely" principle (P8).
- **Mixing cross-scope hints into primary results:** Cross-scope hints must be a separate, clearly flagged output -- never merged into the primary ranked list.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Scope filtering in vector search | Custom pre-filter index | DuckDB join on vector_metadata + nodes table | Already have the metadata; join is fast and correct |
| Scope filtering in lexical search | Custom tantivy pre-filter | Tantivy query parser AND clause with scope field | Tantivy natively supports field-level filtering |
| Multi-scope IN clause | Manual Python list filtering | DuckDB parameterized IN clause | SQL is the right tool for set membership |
| Temporal validity check | Custom date range logic | DuckDB `valid_from <= ? AND (valid_to IS NULL OR valid_to > ?)` | Already used in query_nodes and get_neighborhood |

**Key insight:** Every filtering mechanism already exists in the codebase -- the problem is purely that the candidate generation functions don't pass the filter parameters through to the backends.

## Common Pitfalls

### Pitfall 1: Breaking Backward Compatibility on Scope Parameter Type Change

**What goes wrong:** Changing `scope: Scope | None` to `scope: list[Scope] | None` across the API boundary breaks callers that pass a single `Scope` enum value.
**Why it happens:** The type change from single value to list is a breaking change at every call site.
**How to avoid:** Accept `Scope | list[Scope] | None` at the entry point (MemoryEngine.retrieve(), RetrievalPipeline.retrieve()), normalize to `list[Scope] | None` internally immediately. A single Scope value is wrapped in a list. None means "all scopes" (no filter).
**Warning signs:** Tests that pass a single Scope value start failing with type errors.

### Pitfall 2: Vector Metadata Table Missing Scope Column

**What goes wrong:** Scope filtering for vector search requires knowing which scope a vector belongs to. The `vector_metadata` table has `node_id` and `user_id` but NOT `scope`.
**Why it happens:** Phase 1 designed vector_metadata for user-level scoping only.
**How to avoid:** Two options: (A) Add a `scope` column to `vector_metadata` and index it, requiring a schema migration. (B) JOIN `vector_metadata` with `nodes` table on `node_id` to get scope -- no migration needed but slightly more expensive query. Option (B) is recommended for this phase since the join is local DuckDB (no network latency) and avoids denormalization.
**Warning signs:** Empty vector results when scope filter is applied because the join finds no matches due to orphaned vector entries.

### Pitfall 3: Tantivy Schema Modification Requires Reindex

**What goes wrong:** Adding a `scope` field to the tantivy schema means existing indexed documents don't have the field, so they won't match scope-filtered queries.
**Why it happens:** Tantivy is schema-on-write; existing documents are immutable.
**How to avoid:** Add the `scope` field to the tantivy schema. For existing indexes, the field will be absent from old documents. Tantivy handles missing fields as "no match" -- which means old documents will be excluded from scope-filtered queries. This is acceptable behavior (fail safely: P8). A full reindex could be added as an optional migration step but is not required for correctness since old documents simply won't appear in scope-filtered results.
**Warning signs:** After adding scope field, scope-filtered lexical queries return empty results on existing data until new documents are indexed.

### Pitfall 4: Cross-Scope Hints Doubling Retrieval Cost

**What goes wrong:** Naive cross-scope hint implementation runs the entire candidate generation pipeline twice (once with scope, once without).
**Why it happens:** The obvious approach is "run without filter, compare".
**How to avoid:** Only run the cross-scope hint pass when: (a) scope filter is active, (b) include_cross_scope=True, and (c) primary results exist. For the hint pass, run only vector + lexical (cheapest backends) with a reduced k, then check if any results are outside the primary scope. Graph traversal for hints is unnecessary since graph neighbors of in-scope entities are likely in the same scope.
**Warning signs:** Retrieval latency doubles when scope filter is active.

### Pitfall 5: Temporal Filter Excluding Long-Lived Entity Nodes

**What goes wrong:** Applying strict temporal filtering excludes Entity and Preference nodes that are "always valid" (valid_to=NULL, valid_from=years ago) when the query has a narrow temporal window like "last week".
**Why it happens:** Entity nodes represent persistent concepts, not time-bounded assertions. A temporal filter for "last week" should not exclude the entity "Alice" just because it was created months ago.
**How to avoid:** Recommend exempting certain node types (ENTITY, PREFERENCE) from strict temporal filtering during candidate generation. These types represent persistent knowledge that should always be retrievable. The temporal filter applies to their related facts/decisions/events, not to the entities themselves.
**Warning signs:** Queries like "what did Alice work on last week" return zero results because the Entity node for Alice is excluded by the temporal filter.

## Discretion Recommendations

The following recommendations address the areas left to Claude's discretion in CONTEXT.md.

### Default temporal behavior (no temporal filter provided)

**Recommendation: All-time (no temporal filtering by default)**
- Confidence: HIGH
- Rationale: Consistent with current behavior. Adding a default temporal filter would be a breaking change and would exclude historical facts that are still valid. Temporal filtering should be opt-in, like scope filtering.

### Temporal boundary handling (hard cutoff vs grace window)

**Recommendation: Hard cutoff with `valid_from <= now AND (valid_to IS NULL OR valid_to > now)` semantics**
- Confidence: HIGH
- Rationale: This is already the pattern used in `query_nodes()` and `get_neighborhood()`. A grace window adds complexity with unclear benefit. If needed, the caller can simply expand their temporal window.

### Temporal filtering and node type exemptions

**Recommendation: Exempt ENTITY and PREFERENCE node types from temporal filtering**
- Confidence: MEDIUM
- Rationale: Entities and Preferences are persistent knowledge objects. A query for "what did Alice do last week" should still find the Entity "Alice" -- the temporal filter should apply to the events/facts/decisions about Alice, not to Alice herself. This is the same pattern used by knowledge graph systems where entity nodes are atemporal anchors. The exemption only applies during candidate generation; the temporal filter still applies to all other node types.

### Temporal intent: constrain generation vs re-rank after

**Recommendation: Constrain at candidate generation (pre-filter)**
- Confidence: HIGH
- Rationale: This is the same principle as scope filtering -- pre-filter is required by NSPC-05 and the RFC-0004 philosophy. Post-filter would return expired candidates that waste candidate slots and scoring compute. The temporal window from QueryAnalysis should be forwarded to backends as a WHERE clause, not used for re-ranking after.

### Cross-scope hint token budget allocation

**Recommendation: Separate budget, small and fixed**
- Confidence: MEDIUM
- Rationale: Cross-scope hints should not compete with primary results for token budget. A fixed budget of 512 tokens (or configurable) for cross-scope hints ensures primary results always get their full budget. This is a [HYPOTHESIS] value that should be tunable. The hints are supplementary context, not primary retrieval -- they should be cheap.

## Open Questions

1. **Tantivy Reindex Strategy**
   - What we know: Adding `scope` field to tantivy schema means existing documents lack the field and won't match scope-filtered queries. tantivy handles missing fields as "no match."
   - What's unclear: Whether a full reindex migration should be added to this phase or deferred.
   - Recommendation: Defer full reindex to a separate utility. Accept that scope-filtered lexical queries on pre-migration data return empty results. This is safe degradation per P8 and consistent with how epistemic_type backfill was handled in Phase 3.1.

2. **Cross-Scope Hint Top-N Threshold**
   - What we know: User decision says "high bar, rare hints" -- only results that would rank in top-N ignoring scope.
   - What's unclear: What N should be. Top-3? Top-5? Top-10?
   - Recommendation: Default N=5 as [HYPOTHESIS]. This means a cross-scope result must be in the top 5 unscoped results to appear as a hint. Configurable via PackingConfig.

3. **Vector Metadata Scope Column vs JOIN**
   - What we know: Adding scope to vector_metadata requires migration; JOIN with nodes avoids migration but is slightly slower.
   - What's unclear: Performance impact of the JOIN on large datasets.
   - Recommendation: Use JOIN approach for this phase (no migration needed). The JOIN is local DuckDB, and at MVP scale the performance difference is negligible. If performance becomes an issue, add the scope column as a denormalization optimization later.
