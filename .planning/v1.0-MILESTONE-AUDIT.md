The Revised RFCs likely introduce new or changed requirements not yet captured. A requirements reconciliation pass is recommended before proceeding to Phase 3.

### Integration Issues

**Issue 1 — WriteQueue Bypass (Design Contract Violation)**
- `EntityMerger.find_or_create_entity()` at `entity_merge.py:106` calls `graph_store.create_node()` directly
- `SupersedenceDetector.detect_and_supersede()` at `supersedence.py:164` calls `graph_store.supersede()` directly
- Both bypass WriteQueue; rely on GraphStore's internal `_write_lock` as fallback
- **Affected:** INGE-05
- **Fix:** Inject WriteQueue into EntityMerger/SupersedenceDetector, or wrap calls in `_materialize()`

**Issue 2 — Event.scope Not Persisted**
- `Event` inherits `scope` from `MemoryObject`, but `events` DuckDB table has no `scope` column
- `EventStore._append_sync()` does not write scope; `_row_to_event()` does not read it
- **Affected:** STOR-01, STOR-06
- **Fix:** Add `scope` column to events table, or document that scope is N/A for raw events

**Issue 3 — Scope Not Threadable Through Ingestion**
- `IngestionPipeline.ingest()` has no `scope` parameter
- All extracted MemoryNodes are hardcoded to `Scope.PERSONAL`
- `MemoryEngine.store()` (lower-level) does accept scope
- **Affected:** STOR-06, INGE-04
- **Fix:** Add `scope: Scope = Scope.PERSONAL` parameter and thread through to materialization

**Issue 4 — OpenAI Embedding asyncio.run() Fragility**
- `OpenAIEmbeddingProvider.embed()` at `embedding.py:206` uses `asyncio.run()`
- Works only when called from thread pool (via `to_thread`), not from async context directly
- Currently safe because VectorIndex always calls via `to_thread`, but fragile for future callers
- **Affected:** INGE-03
- **Fix:** Restructure to use dedicated thread event loop or `nest_asyncio`

## Revised RFC Gap Analysis

**The Revised RFC suite (`docs/RevisedRFCs/`, RFC-0000 through RFC-0014) was added after Phases 1-2 were planned, built, and verified.** The current REQUIREMENTS.md was derived from the original spec documents (`docs/`). The Revised RFCs represent a scientifically audited replacement of the original suite with significant changes.

### Key Revised RFC concepts not captured in current REQUIREMENTS.md:

| Revised RFC | Concept | Impact on Built Code |
|-------------|---------|---------------------|
| RFC-0001 (Core Data Model) | Revised type system, terminology | May change NodeType/EdgeType enums, model fields |
| RFC-0003 (Epistemic State Model) | Revised lifecycle semantics | May affect LifecycleState transitions, confidence scoring |
| RFC-0004 (Namespace and Scope Isolation) | Full namespace isolation spec | Current scope implementation is minimal (PERSONAL hardcoded in ingestion) |
| RFC-0005 (Hybrid Retrieval) | Revised retrieval pipeline spec | Phase 3 will consume this — may affect retrieval API contracts established in Phase 1 |
| RFC-0007 (Decay and Forgetting) | Decay model for salience | Phase 5 scope — may require salience field changes in Phase 1 models |
| RFC-0008 (Confidence Evolution) | Confidence reinforcement model | May affect how confidence scores are tracked on nodes/edges |
| RFC-0013 (Intent and Goal Memory) | Replaces emotional tracking with intent/goal tracking | New node types or metadata may be needed |
| Suite-wide | `[HYPOTHESIS]` markers on all unvalidated parameters | Requires benchmark gates before parameter values are promoted from defaults |
| Suite-wide | Scoped determinism (strict at storage, best-effort at extraction) | Already aligned with implementation, but not formalized in REQUIREMENTS.md |

### Recommendation

Before proceeding to Phase 3, reconcile REQUIREMENTS.md against the Revised RFC suite:
1. Read each RFC relevant to completed phases (RFC-0001 through RFC-0004)
2. Identify new or changed requirements
3. Add them to REQUIREMENTS.md
4. Create gap-closure phases (via `/gsd:insert-phase`) for any delta in built code
